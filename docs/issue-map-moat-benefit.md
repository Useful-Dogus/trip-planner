# 이슈 맵 (붙여넣기용) — 해자와 사용자 이익 향상

> 이 로드맵의 척추는 두 레버다.
> **(A) 사용자 이익** — "내 후보 더미 → 현장에서 안 깨지는 일정". 사용자가 한 문장으로 말할 수 있고 경쟁 제품으론 못 얻는 이득.
> **(B) 제품의 해자** — 신뢰(정확도) · 독점 데이터 루프 · 전환비용. 경쟁자가 코드로 못 베끼는 방어선.
> 모든 이슈는 A 또는 B를 측정 가능하게 올려야 한다. 둘 다 안 올리면 보류.
>
> 사용법: **에픽 이슈**를 먼저 만들고, **상세 이슈**들을 만든 뒤 에픽 본문 `#__`에 번호를 채운다(또는 GitHub Sub-issues로 연결).
> 근거: [docs/strategy-moat-and-direction.md](strategy-moat-and-direction.md)

각 이슈 태그: `[A]` 사용자 이익 · `[B:신뢰]` `[B:데이터]` `[B:전환]` 해자 종류. (둘 다면 병기)

---

## ⭐ 에픽 (트래킹 이슈)

**제목:** `[Epic] 해자와 사용자 이익 향상 로드맵 (research pile → 안 깨지는 일정)`

**본문:**

```markdown
## 배경
이 로드맵은 두 레버만 올린다.
- (A) 사용자 이익: 내가 모은 후보 더미를 추려서 현장에서 안 깨지는 일정으로 만드는 다리.
- (B) 제품의 해자: 신뢰(정확도)·독점 데이터 루프·전환비용.
모든 이슈는 A 또는 B를 측정 가능하게 올려야 한다.
전략 근거: docs/strategy-moat-and-direction.md

## 레버 A — 사용자 이익을 직접 키우는 작업 (research pile → 안 깨지는 일정)
- [ ] #__ [A][B:신뢰] 일정 뷰에 확정 항목 간 이동시간·거리 표시 (좌표 기반)
- [ ] #__ [A][B:신뢰] 비효율 동선 구간 시각적 경고
- [ ] #__ [A][B:데이터] 후보→보류→확정→탈락 상태 전환 1~2클릭으로 단축
- [ ] #__ [A] 확정 항목이 일정(날짜)으로 떨어지는 흐름 개선
- [ ] #__ [A] 빈 상태·랜딩 카피를 "사용자 이익 한 문장"으로 교체
- [ ] #__ [A][B:데이터] 내 후보 더미 기반 자동 일정 제안 (플래너 에이전트 v1)
- [ ] #__ [A] 개인 제약(반드시 갈 곳·예산) 반영 일정 생성

## 레버 B — 해자를 쌓는 작업 (경쟁자가 못 베끼게)
- [ ] #__ [B:신뢰][A] 영업시간·휴무일 데이터 연동 및 일정 충돌 경고
- [ ] #__ [B:신뢰][A] 예약 마감·마지막 입장 시각 표시
- [ ] #__ [B:데이터] 항목 보류/탈락 사유 한 줄 캡처
- [ ] #__ [B:데이터] 여행 후 만족도 피드백 캡처
- [ ] #__ [B:데이터][B:전환] 항목별 그룹 투표·선호 표시
- [ ] #__ [B:전환] 여행 단일 진실 공급원(시스템 오브 레코드) 강화 및 전환비용 설계

## 실행 순서 (선행조건 기준 — 라벨이 아니라 의존성·가치로 정렬)
1. 이익 즉효·비용 낮음·선행조건 없음: 이동시간 표시 → 동선 경고 → triage 단축 → 일정 배치 흐름 → 이익 카피
2. 해자 적립 시작(저비용): 사유 캡처
3. A·B 동시(외부 데이터 필요): 영업시간 검증 → 예약 마감 표시
4. 데이터 위에서 이익 크게: 자동 일정 → 제약 반영
5. 사용자 스케일 필요(2번째 사용자/사용 누적 시): 그룹 투표 → 사후 피드백 → 시스템 오브 레코드

## 완료 기준 (에픽)
- A: "이거 덕분에 일정 짜기가 쉬워졌고, 현장에서 안 깨진다"가 참.
- B: 사유·피드백·결정 이력이 쌓이고, 신뢰 검증이 동작.
```

라벨 제안: `epic`, `roadmap`

## 상세 이슈

> 각 이슈 제목 앞 태그는 올리는 레버. 본문 첫 줄에 "레버:"로 명시.

### 1. 일정 뷰에 확정 항목 간 이동시간·거리 표시 (좌표 기반) `[A][B:신뢰]`
```markdown
부모 에픽: #__
레버: A(안 깨지는 일정) + B:신뢰(정확도 해자의 1차)

## 목표
연속한 확정 항목 사이의 예상 이동시간·거리를 표시해 동선이 한눈에 보이게 한다.

## 작업
- 좌표(lat/lng) 기반 거리 계산 (외부 API 없이 시작, 기존 DistanceSeparator 확장)
- 일정 뷰 항목 사이에 "이동 ~분 / ~km" 표시
- 좌표 누락 항목 graceful 처리

## 수용 기준
- [ ] 같은 날 확정 항목 2개 이상이면 사이에 이동 정보 노출
- [ ] 좌표 누락 시 깨지지 않음
- [ ] 모바일·데스크탑 양쪽 표시
```
라벨: `lever-a`, `lever-b-trust`, `schedule`

### 2. 비효율 동선 구간 시각적 경고 `[A][B:신뢰]`
```markdown
부모 에픽: #__
레버: A + B:신뢰

## 목표
왕복/역행 등 비효율 구간을 사용자가 인지하게 한다.

## 수용 기준
- [ ] 이동시간 임계값 초과 구간에 경고 표시
- [ ] 정보성(차단 아님), 디자인 가이드 토큰 사용
```
라벨: `lever-a`, `lever-b-trust`, `schedule`

### 3. 후보→보류→확정→탈락 상태 전환 1~2클릭으로 단축 `[A][B:데이터]`
```markdown
부모 에픽: #__
레버: A(적은 노력으로 추리기) + B:데이터(사용 유발 → 결정 이력 적립)

## 목표
triage가 매일 동작이므로 상태 변경을 최소 클릭으로.

## 수용 기준
- [ ] 카드/행에서 상태 변경 1~2클릭
- [ ] list/map/schedule 6조합에서 동등 도달 (Basics-First #4)
- [ ] 낙관적 업데이트
```
라벨: `lever-a`, `lever-b-data`, `ux`

### 4. 확정 항목이 일정(날짜)으로 떨어지는 흐름 개선 `[A]`
```markdown
부모 에픽: #__
레버: A(다리 연결)

## 목표
확정 시 자연스럽게 날짜·시간 배정으로 이어지는 흐름.

## 수용 기준
- [ ] 확정 전환 시 날짜 배정 진입점 제공
- [ ] 날짜 미배정 확정 항목을 일정 뷰에서 식별 가능
```
라벨: `lever-a`, `schedule`

### 5. 빈 상태·랜딩 카피를 "사용자 이익 한 문장"으로 교체 `[A]`
```markdown
부모 에픽: #__
레버: A(이익의 가독성)

## 목표
"왜 이걸 쓰는가 = 어떤 이득"을 한 문장으로. 기능 나열·협업 강조 카피 제거.

## 수용 기준
- [ ] 랜딩/빈 상태 카피가 사용자 이익 한 문장 반영
- [ ] 카피가 약속한 동작이 실제 가능 (Basics-First #3)
```
라벨: `lever-a`, `copy`

### 6. 항목 보류/탈락 사유 한 줄 캡처 `[B:데이터]`
```markdown
부모 에픽: #__
레버: B:데이터(독점 데이터 루프의 시작점)

## 목표
"이거 왜 뺐더라"를 남겨 결정 이력을 자산화한다. (소비는 후순위, 캡처를 먼저)

## 작업
- items 스키마에 사유 텍스트 컬럼 추가 (nullable)
- 보류/탈락 전환 시 선택적 한 줄 입력

## 수용 기준
- [ ] 사유 입력·표시·수정 가능 (선택 입력)
- [ ] 기존 항목 마이그레이션 무해
```
라벨: `lever-b-data`, `data`

### 7. 영업시간·휴무일 데이터 연동 및 일정 충돌 경고 `[B:신뢰][A]`
```markdown
부모 에픽: #__
레버: B:신뢰(정확도 해자의 본체) + A(안 깨지는 일정)
의존: #1(이동시간 표시), #6(스키마 확장 패턴)

## 목표
방문 시각이 영업시간 밖/휴무일이면 경고해 "현장에서 안 깨진다"를 사실로.

## 작업
- 영업시간·휴무 데이터 소스 결정 (google_place_id 활용 또는 수동 입력 폴백)
- items에 opening_hours/closed_days 캐시 컬럼
- 일정 뷰에서 방문 시각 vs 영업시간 대조, 위반 시 경고 배지
- 데이터 없음/불확실은 "확인 필요"로 구분 (거짓 단정 금지)

## 수용 기준
- [ ] 휴무·영업외 시각 방문에 경고
- [ ] 데이터 없는 항목은 "정보 없음" 처리(오탐 방지)
- [ ] 외부 데이터 실패 시 일정 뷰 안 깨짐
```
라벨: `lever-b-trust`, `lever-a`, `data`

### 8. 예약 마감·마지막 입장 시각 표시 `[B:신뢰][A]`
```markdown
부모 에픽: #__
레버: B:신뢰 + A
의존: #7

## 목표
예약 마감·마지막 입장을 담고 일정상 위반 시 알린다.

## 작업
- items에 last_entry_time / reservation_deadline 필드
- 항목 폼 입력 UI
- 도착 추정 시각 > 마지막 입장이면 경고

## 수용 기준
- [ ] 입력·표시·수정 가능(선택 입력)
- [ ] 위반 시 경고, 값 없으면 경고 없음
```
라벨: `lever-b-trust`, `lever-a`

### 9. 내 후보 더미 기반 자동 일정 제안 (플래너 에이전트 v1) `[A][B:데이터]`
```markdown
부모 에픽: #__
레버: A(적은 노력) + B:데이터(독점 후보·결정 데이터를 입력으로)
의존: #1, #4, #7

## 목표
범용 추천이 아니라 **내 후보 + 제약** 위에서만 도는 자동 일정. (에이전트를 앱 envelope 안에)

## 작업
- 입력: 확정/후보 항목·좌표·영업시간·날짜 범위·basecamp
- 동선·영업시간 고려한 일자별 배치 "초안" 생성 (자동 확정 금지)
- 1차 규칙 기반(이동시간 최소 + 영업시간 충족), LLM은 후속

## 수용 기준
- [ ] 후보 N개 → 일자별 초안 생성
- [ ] 결과가 영업시간·휴무 위반 안 함
- [ ] 사용자가 초안 수정·일부 수락 가능
- [ ] 외부 장소를 새로 주입하지 않음(내 후보 한정)
```
라벨: `lever-a`, `lever-b-data`, `agent`

### 10. 개인 제약(반드시 갈 곳·예산) 반영 일정 생성 `[A]`
```markdown
부모 에픽: #__
레버: A
의존: #9

## 목표
우선순위("반드시")와 예산을 자동 일정 제약으로 반영.

## 수용 기준
- [ ] "반드시" 항목 100% 포함
- [ ] 예산 초과 시 사유와 함께 표시
- [ ] 제약 불충족을 조용히 누락하지 않고 설명
```
라벨: `lever-a`, `agent`

### 11. 여행 후 만족도 피드백 캡처 `[B:데이터]`
```markdown
부모 에픽: #__
레버: B:데이터(루프 강화)
의존: 사용·여행 완료 누적

## 목표
다녀온 항목의 "실제로 좋았나"를 남겨 다음 추천의 데이터로.

## 수용 기준
- [ ] 항목별 사후 피드백 입력·표시(선택)
- [ ] 후속 추천에 쓸 수 있는 형태로 저장
- [ ] 입력 강요 없음
```
라벨: `lever-b-data`, `data`

### 12. 항목별 그룹 투표·선호 표시 `[B:데이터][B:전환]`
```markdown
부모 에픽: #__
레버: B:데이터 + B:전환
선행조건: 2번째 실사용자 / 멀티유저·공유(기존 #108 계열)

## 목표
일행이 후보 선호를 가볍게 표시해 추리기 결정을 집단으로 가시화.

## 수용 기준
- [ ] 멤버별 1항목 1투표 토글
- [ ] 집계 실시간 반영(낙관적 업데이트)
- [ ] viewer는 투표 불가 (RLS 일관)
```
라벨: `lever-b-data`, `lever-b-switching`, `collab`

### 13. 여행 단일 진실 공급원 강화 및 전환비용 설계 `[B:전환]`
```markdown
부모 에픽: #__
레버: B:전환(시스템 오브 레코드)
선행조건: 사용자·공유 수요 확인

## 목표
후보·결정·예약·메모·동행자를 한 여행의 단일 진실 공급원으로 모아 전환비용을 만든다.

## 작업
- 여행 단위 export/백업 (소유권 신뢰 → 잔존율↑)
- 예약·문서 첨부를 여행에 귀속
- 동행자 활동이 한 곳에 모이는 뷰

## 수용 기준
- [ ] 한 여행 핵심 데이터를 단일 위치에서 조회·export
- [ ] 멤버 변화에도 데이터 귀속 일관
- [ ] Basics-First(컨텍스트 표시·CRUD 대칭) 통과
```
라벨: `lever-b-switching`, `collab`
