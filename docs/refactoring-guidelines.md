# Refactoring Guidelines — 리팩토링 규율 (도구 중립)

> 이 문서는 **에이전트 기반 개발 중간중간** 수행하는 tidy-first·구조 개선·아키텍처 정리의 단일 기준이다.
> Claude·Codex·Gemini 등 어떤 코딩 에이전트를 쓰든 **같은 규칙**을 따르도록 도구 중립으로 작성한다.
>
> - 진입점: 루트 [`AGENTS.md`](../AGENTS.md) (표준), [`CLAUDE.md`](../CLAUDE.md) (`@import`).
> - 자매 문서: [design-guidelines.md](design-guidelines.md) = *메커닉*, [taste-for-waypost.md](taste-for-waypost.md) = *판단*, CLAUDE.md 의 **Basics-First Rule** = *실천 게이트*. 리팩토링은 그 셋을 **유지보수 가능한 상태로 떠받치는 기반 작업**이다.
>
> **담는 것은 거의 모든 현대 개발자가 동의하는 합의된 원칙뿐이다.** 유행하는 패턴·특정 프레임워크 취향·논쟁적 스타일은 넣지 않는다. 출처: Kent Beck *Tidy First?* (2023), Martin Fowler *Refactoring* 2e (2018), Robert C. Martin *Clean Architecture* (2017) 중 광범위 합의 항목.

---

## 0. 한 줄

> **리팩토링은 동작을 바꾸지 않고 구조만 개선하는 것이다.** 동작 변경과 절대 섞지 않는다. 작게 쪼개고, 테스트(또는 검증)를 초록으로 유지하며, 별도로 커밋한다.

---

## 1. 두 개의 모자 (Two Hats) — 가장 중요한 규칙

한 번에 **하나만** 한다. 모자를 바꿔 쓰되 동시에 쓰지 않는다.

| 모자 | 하는 일 | 커밋 |
| --- | --- | --- |
| **구조 변경 (structural)** | 이름 바꾸기, 추출, 이동, 인라인. **동작 불변.** | `refactor:` / `style:` |
| **동작 변경 (behavioral)** | 기능 추가·버그 수정. 입출력이 달라짐. | `feat:` / `fix:` |

- 두 종류 변경을 **한 커밋·한 diff 에 섞지 않는다.** 리뷰어(사람·에이전트)가 "이 줄은 안전한 정리인가, 동작이 바뀐 곳인가"를 한눈에 구분할 수 있어야 한다.
- 기능 작업 중 정리할 곳이 보이면: **먼저 구조 변경만 따로 커밋 → 그 위에 기능 변경.** (Tidy First)
- 구조 변경 PR 은 diff 가 커도 **읽기 쉽고 위험이 낮아야** 한다. 그렇지 않다면 동작이 섞여 들어간 것이다.

---

## 2. Tidy First — 언제 정리하는가

정리(tidying)는 작은 구조 개선이다. **타이밍**이 핵심이다.

- **지금 정리 (먼저)**: 곧 손댈 코드가 지저분해서, 정리하면 **이번 변경이 더 쉬워질 때.** "이해하려고 읽다가 깨달은 것"을 즉시 코드에 반영한다.
- **나중에 정리**: 지금 변경과 무관한 곳. 발견은 기록(이슈·TODO)하되, 현재 PR 범위를 넓히지 않는다.
- **안 함**: 다시 안 건드릴 코드, 또는 정리 비용이 이득보다 큰 경우.

대표적 tidying (모두 동작 불변, 수 분 내):
- 가드 절(guard clause)로 중첩 if 펴기
- 죽은 코드·도달 불가 분기 삭제
- 설명적 이름으로 변경, 매직 넘버 → 상수
- 의미 단위로 빈 줄·헬퍼 추출
- 한 군데서만 쓰는 과한 추상화 인라인

> **규모 규칙:** tidying 은 작아야 한다. 한 정리가 수십 파일·수백 줄이 되면 그건 tidying 이 아니라 별도 계획·별도 PR 이 필요한 **아키텍처 변경**이다 (§5).

---

## 3. 안전하게 — 작은 걸음 + 초록 유지

- **테스트(또는 명확한 검증 경로)가 초록인 상태에서 시작**하고, 각 작은 걸음마다 다시 초록으로 되돌린다. 테스트가 없으면 리팩토링 전에 **특성화 테스트(characterization test)** 부터 추가하거나, 최소한 수동 검증 절차를 명시한다.
- 한 번에 **하나의** 변환만. 추출하고 → 검증 → 커밋, 그다음 이름 바꾸고 → 검증 → 커밋. 묶어서 몰아치지 않는다.
- 가능하면 **자동 리팩토링 도구**(IDE rename/extract, codemod)를 손수정보다 우선한다 — 실수 여지가 적다.
- 커밋을 자주, 가역적으로. 막히면 마지막 초록 커밋으로 되돌릴 수 있어야 한다.

---

## 4. 무엇을 고치는가 — 합의된 코드 냄새

아래는 거의 모든 진영이 동의하는 신호다. 보이면 정리 후보:

- **중복 (Duplicated Code)** — 단, §6 의 Rule of Three 를 지켜 성급히 추상화하지 않는다.
- **긴 함수 / 큰 클래스 / 긴 파라미터 목록** — 하나의 책임 단위로 쪼갠다.
- **깊은 중첩 / 복잡한 조건** — 가드 절, 다형성, 조기 반환으로 평탄화.
- **죽은 코드 · 추측성 일반화(speculative generality)** — 지금 안 쓰면 지운다 (YAGNI).
- **나쁜 이름** — 의도를 드러내지 못하는 이름. 가장 값싸고 효과 큰 개선.
- **흩어진 변경(shotgun surgery) / 뒤엉킨 책임(divergent change)** — 한 가지 이유로 바뀌는 것들을 한곳에 모은다.
- **누설된 추상화 / 부적절한 친밀(feature envy)** — 데이터와 그걸 쓰는 로직을 가까이 둔다.

---

## 5. 아키텍처 변경 — 보수적으로, 합의된 부분만

큰 구조 변경은 가치 있지만 위험하다. **논쟁적 패턴은 피하고**, 다음 합의 원칙만 보수적으로 적용한다.

- **관심사 분리 (SoC):** UI / 도메인 로직 / 데이터 접근을 섞지 않는다. 이 리포에선 `app|components` (UI) ↔ `lib` (도메인·어댑터) ↔ `supabase` (저장소) 경계를 흐리지 않는다.
- **의존성 방향:** 안정적·추상적인 쪽으로 의존한다. 도메인 로직이 특정 벤더 SDK(예: Supabase 클라이언트)를 직접 알지 않도록 `lib/*` 어댑터 뒤에 둔다. (메모리의 "어댑터 원칙"과 일치.)
- **단일 책임(SRP):** 모듈/함수가 바뀌는 이유는 하나여야 한다.
- **YAGNI > DRY > 성급한 추상화:** 추상화는 **중복이 실제로 3번** 나타난 뒤에 (§6). 미래를 위한 일반화 레이어를 미리 깔지 않는다.

### 5.1 아키텍처 변경은 반드시 격리한다

- **기능 PR 안에서 큰 구조 변경을 동시에 하지 않는다.** 별도 PR, 별도 커밋 시퀀스.
- 변경 전 **현재 동작을 잠그는 테스트/검증 경로**를 먼저 확보한다.
- 한 번에 한 경계만 옮긴다. "전부 갈아엎기"는 금지 — 작은 이주(strangler-fig 식)로 점진 교체.
- 되돌릴 계획이 없는 대규모 재작성은 제안하지 않는다. 제안해야 한다면 사용자에게 **명시적으로 알리고 승인**을 받는다.

---

## 6. 하지 말 것 (성급함 방지)

- **Rule of Three:** 중복은 두 번까진 참는다. **세 번째**에 추출한다. 한 번 본 패턴으로 추상화하지 않는다 (AHA — Avoid Hasty Abstractions).
- **동작을 바꾸면서 "리팩토링"이라 부르지 않는다.** 그건 기능/수정이다.
- **요청 범위를 넘는 정리를 몰래 끼워넣지 않는다.** 사용자가 A 를 고쳐달라 했는데 무관한 B·C·D 를 함께 갈아엎지 않는다 — 발견은 보고하고 분리한다.
- **테스트 없는 코드를 검증 없이 대규모 변경하지 않는다.**
- **이름만 다르고 똑같은 추상화를 새로 만들지 않는다** — 기존 유틸·패턴을 먼저 찾는다.

---

## 7. 에이전트 작업 규율 (실행 체크리스트)

코딩 에이전트가 작업 도중 리팩토링할 때 매번 자기검열한다:

1. **모자 확인** — 지금 하는 게 구조 변경인가 동작 변경인가? 섞고 있지 않은가? (§1)
2. **타이밍** — 이 정리가 *이번* 변경을 쉽게 하는가? 아니면 범위 확장인가? (§2)
3. **크기** — tidying(수 분) 인가 아키텍처(별도 PR) 인가? 후자면 격리·승인. (§5.1)
4. **안전망** — 검증 경로(테스트/실행)가 있는가? 없으면 먼저 확보. (§3)
5. **커밋 분리** — 구조 변경은 `refactor:`/`style:` 로, 동작 변경과 별도 커밋. (§1)
6. **범위 정직성** — 요청 범위 밖 정리는 같이 끼워넣지 말고 **보고**한다. (§6)
7. **기존 것 재사용** — 새 추상화 만들기 전에 `lib/`·기존 패턴을 검색한다. (§6)

> 의심스러우면 **덜** 한다. 작고 분리되고 가역적인 변경이, 크고 영리한 변경보다 거의 항상 낫다.
