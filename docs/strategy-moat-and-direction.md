# Trip Planner — 전략: 해자와 사용자 이익을 높이는 계획

작성일: 2026-06-22 (개정)
참고: SignalFire, "The 4 arguments for the death of software" (2026-06-03)

## 0. 계획의 핵심 — 두 개의 레버

이 계획이 끌어올리려는 것은 단 두 가지다.

1. **사용자 이익** — 사용자가 *한 문장으로 말할 수 있고, 경쟁 제품으론 못 얻는* 구체적 이득. (채택·잔존의 이유)
2. **제품의 해자** — 경쟁자가 코드로 *못 베끼는* 구조적 방어선: 신뢰(정확도)·독점 데이터 루프·전환비용. (가격·복제 방어의 이유)

이익은 *왜 쓰는가*를, 해자는 *왜 못 베끼는가*를 만든다. 둘 다 있어야 산다. **모든 작업은 이 둘 중 최소 하나를 측정 가능하게 올려야 한다. 아니면 보류.** 이게 모든 우선순위의 기준이다. (작업 순서로 나오는 "혼자 쓸 때 가치 나는 것 먼저"는 이 레버에서 *도출되는 결과*이지 목표가 아니다.)

## 1. 레버 A — 사용자 이익

### 정의 (한 문장으로 말할 수 있는 이득)

> **"내가 모은 후보 더미를, 버리지 않고 추려서, 현장에서 안 깨지는 하루 일정으로 만든다."**

이익의 실체 = 세 가지가 동시에: ① 내가 한 리서치를 *잃지 않고*, ② *적은 노력으로* 추려서, ③ 현장에서 *안 깨지는* 일정이 된다. "예쁜 일정표"가 아니라 **research pile → workable itinerary 의 다리**가 이익의 핵심이다.

### 왜 경쟁 제품으론 못 얻는가

| 경쟁 제품 | 잘하는 것 | 이 이익을 못 주는 지점 |
|-----------|-----------|------------------------|
| 트리플 | *그들의* 콘텐츠로 자동 일정 | 내가 직접 모은 후보에서 출발하지 못함 (자기 DB 기반) |
| Wanderlog | 리스트 + 지도 | 위시리스트와 일정이 분리, 추리는 과정이 흐름이 아님 |
| Google My Maps | 무료 핀 | 추리기·일정·동선/영업시간 검증 없음 |
| 스프레드시트 | 자유로운 정리 | 지도·동선·영업시간 없음 → "안 깨지는지"를 못 봄 |
| 범용 GPT | 즉석 일정 | 내 후보 더미를 안 들고 있고, 영업시간·이동시간을 환각 |

### 이익을 "올린다"의 의미 + 측정 지표

이 다리를 **더 적은 노력으로**(추리기·배치 자동화) + **더 안 깨지게**(동선·영업시간·마감 검증) 만드는 모든 것이 레버 A를 올린다. "측정 가능"을 표방하므로 지표를 정의한다:

- **노력:** 후보→확정까지 평균 클릭/시간, triage 1건당 클릭 수
- **안 깨짐:** 확정 일정의 제약 위반(휴무·영업외·마지막 입장 초과·과도 동선) 건수 / 여행
- **잃지 않음:** 보류 후 재검토되어 확정으로 살아난 후보 비율

각 이슈의 수용 기준은 boolean 체크가 아니라 위 지표 중 하나의 **방향(↑/↓)** 을 명시해야 한다.

## 2. 레버 B — 제품의 해자 (essay의 moat 3종)

| 해자 | 여행 버전 | 기반 자산 | 베끼기 어려운 이유 |
|------|-----------|-----------|--------------------|
| 고정확도(에러 버짓↓) | "현장에서 안 깨진다" — 영업시간·휴무·이동시간·마감 검증 | 좌표 + 장소 데이터 + 검증 규칙 | 범용 LLM이 가장 자주 틀리는 영역. 신뢰는 코드 패리티로 안 됨 |
| 독점 데이터 루프 | 후보·보류/탈락 사유·선호·사후 만족도 | **상태 기반 단일 테이블에 쌓이는 결정 이력** | 같은 파운데이션 모델을 띄워도 내 결정 데이터는 복제 불가 |
| 전환비용(시스템 오브 레코드) | 한 여행의 모든 후보·결정·예약·동행자 | 멀티유저·공유 | 옮기려면 데이터+관계 전체를 옮겨야 함 |

**핵심:** 우리의 "상태 기반 단일 테이블"은 레버 A(추리기 흐름=이익)와 레버 B(결정 데이터=해자)를 *동시에* 떠받친다. 여기가 전략의 중심.

### 해자가 되려면 — 두 가지 조건 (검토에서 잡힌 함정)

**① 데이터는 "루프"가 닫혀야 해자다. 캡처만 하면 그냥 로그다.**
사유·피드백을 모으는 것(캡처)만으로는 복제 불가 자산이 안 된다. *그 데이터가 다시 제품 출력(추천·자동 일정)을 개선*하고, 더 나은 출력이 다시 사용을 늘려 데이터를 더 모으는 고리가 닫혀야 한다. → **규칙: 캡처 작업(사유·피드백)은 반드시 그것을 소비하는 작업(자동 일정·추천)과 의존성으로 연결한다.** 소비처 없는 캡처는 우선순위를 못 받는다.

**② 정확도는 "공용 소스"로 채우면 해자가 아니다 — 패리티다.**
영업시간·이동시간을 Google 등 *누구나 같은 곳*에서 가져오면 경쟁자도 똑같이 한다(해자 0). 더구나 좌표 직선거리는 실제 이동과 어긋나 *신뢰를 깎는다*. 신뢰가 해자가 되는 지점은 **우리 사용자가 현장에서 검증·수정한 데이터가 쌓일 때**(crowd-correction) 또는 *실측 라우팅*을 쓸 때다. → 공용 소스 표시는 레버 A(편의)로, *사용자 검증 데이터 축적*은 레버 B(신뢰 해자)로 분리해 본다. 단순 직선거리 표시는 `[B:신뢰]`가 아니라 `[A]`다.

## 3. 진단 — 지금은 두 레버 모두 약하다 ("dead zone")

essay의 논리를 그대로 적용하면, 장소 리스트+지도+타임라인은 코드가 싼 시대에 가장 얇은 해자(BI·콘텐츠 생성 범주)이고, 저위험이라 범용 에이전트에 대체되기도 쉽다. 즉 **현재 이익도 "GPT로도 되는데?" 수준, 해자도 거의 0.** 그래서 트리플·Wanderlog 기능을 그대로 따라가는 건 답이 아니다 — 레버 A·B를 올리는 작업만 한다.

## 4. 작업을 두 레버로 채점

| 작업 | 올리는 이익(A) | 쌓는 해자(B) | 비용 | 선행조건 |
|------|:---:|:---:|:---:|----------|
| 이동시간·거리 표시(직선) | ◎ 편의 | — *공용/직선은 해자 아님* | 낮음 | 없음 |
| 비효율 동선 경고 | ◎ | — | 낮음 | 이동시간 |
| triage 1~2클릭 / 일정 배치 흐름 | ◎ 적은 노력 | ○ 사용 유발(간접) | 낮음 | 없음 |
| 보류/탈락 사유 캡처 | ○ 기억 | ◎ 데이터 루프(입력) | 낮음 | 없음 |
| 사후 만족도 피드백 | △ | ◎ 데이터 루프(입력) | 낮음 | 사용·여행 완료 |
| 영업시간·휴무 *검증·수정 축적* | ◎ 안 깨짐 | ◎ 신뢰(crowd-correction) | 중 | 좌표 |
| 예약 마감·마지막 입장 | ◎ | ○ 신뢰 | 중 | 영업시간 패턴 |
| 자동 일정(에이전트) — *사유·피드백 소비* | ◎ 적은 노력 | ◎ 데이터 루프(소비, 고리 닫힘) | 높음 | 사유·피드백·이동·영업시간 |
| 제약(필수·예산) 반영 | ○ | ○ | 중 | 에이전트 |
| 그룹 투표·선호 | △ | ○ 데이터 | 중 | **2번째 사용자** |
| 시스템 오브 레코드·전환비용 | △ | ◎ 전환비용(관계·예약 얽힘) | 높음 | **사용자·공유 수요** |

◎ 큼 / ○ 보통 / △ 사용자 스케일 있어야 가치 / — 해자 아님(패리티)
*주의: 자동 일정은 사유·피드백을 입력으로 소비할 때만 데이터 루프를 닫는다. 그 데이터가 없으면 "내 후보만 쓰는 범용 배치기"가 되어 해자가 0이다(§6 위반).*

## 5. 우선순위 (채점에서 도출)

원칙: **(이익 + 해자) / 비용** 이 높고 선행조건이 적은 것부터. 두 변형 규칙을 명시한다 — (a) *데이터 루프 입력(사유·피드백) 캡처는 복리로 쌓이므로 비용이 낮으면 일찍 시작*한다(검토 지적 반영: 단기 A에만 치우치지 않기), (b) *루프를 닫는 소비 작업(자동 일정)은 그 입력이 쌓인 뒤*에 온다.

1. **데이터 루프 입력 + 즉효 이익을 함께 착수** — (a) 결정 사유 캡처 + 사후 피드백(저비용, 복리 적립 시작), (b) 이동시간·동선 표시 + triage/배치 흐름(A를 가장 싸게 크게). 이 둘은 동시 진행.
2. **신뢰 해자 — 검증·수정 데이터 축적** — 영업시간·마감 검증을 *공용 소스 표시*에서 멈추지 말고 *사용자 현장 수정을 받는 형태*로. A·B 동시 ◎.
3. **루프를 닫는 자동 일정(에이전트)** — 1의 사유·피드백 + 2의 검증 데이터를 **입력으로 소비**. 이때 비로소 데이터가 로그가 아니라 해자가 됨.
4. **제약 반영 일정** — 에이전트 위.
5. **그룹 투표·전환비용** — 2번째 사용자/공유 수요가 실제로 생길 때.

## 6. 하지 말 것 (두 레버 어느 쪽도 안 올림)

- 범용 가이드/추천 콘텐츠 경쟁 — 이익 미미, 해자 얇음.
- "기능 빨리 낸다"를 정체성으로 — 이제 table stakes.
- OTA 예약·결제 정면 진입 — 우리 해자 아님.
- 자동 일정을 *범용 추천*으로 — 독점 데이터(레버 B)를 안 쓰면 대체당함.
- 사용자 없는 단계에서 협업·전환비용 선투자 — 지금 가치 0.

> essay의 결론: "실제 회사가 되는 기준이 막 올라갔다." 장소 리스트+지도는 누구나 만든다. 방어선은 *사용자가 한 문장으로 말하는 이익*과 그걸 떠받치는 *신뢰·데이터·전환비용*에 있다. 다행히 상태 기반 모델이 그 둘의 씨앗을 쥐고 있다 — 이제 두 레버를 측정 가능하게 올리는 일만 남았다.
