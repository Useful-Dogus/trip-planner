# 011 — 체인점 이름 기반 그룹핑

## 배경

여행 리서치 중 동일 체인의 여러 지점을 목록에 등록해두는 사용 패턴이 있다.
예: Trader Joe's를 Chelsea·UWS·Union Square 세 곳 모두 리서치 목록에 올려두고, 실제 방문은 한 곳만 결정.

기존에는 `is_franchise` + `branches(jsonb)` 방식으로 하나의 부모 아이템 안에 지점을 묶었으나, 이 방식은 개별 지점에 대한 독립적인 상태·메모·예약 관리가 불가능했다.
DB 마이그레이션을 통해 모든 지점을 개별 아이템으로 전환 완료.

이제 프론트엔드에서 **이름이 같은 아이템을 자동으로 그룹핑**하여 시각적으로 묶어주는 UX가 필요하다.

## 목표

- 리서치 목록에서 같은 이름의 아이템이 자동으로 하나의 그룹 카드로 묶인다
- 개별 아이템의 독립성(상태, 메모, 예약 등)은 유지된다
- 인라인 이름 편집으로 체인 정렬이 빠르게 가능하다

## 범위

### 포함

- `ItemList` 내 이름 기반 그룹핑 렌더링
- `GroupCard` 신규 컴포넌트 (헤더 + BranchRow 목록)
- `ItemCard` 인라인 이름 편집 (더블클릭)
- `GroupCard` 헤더 인라인 이름 편집 (전체 지점 일괄 rename)
- 필터·정렬과의 상호작용
- 지도 뷰 연동 (핀 클릭 → 그룹 펼침)
- `is_franchise`/`branches` 관련 기존 코드 제거

### 제외

- 모바일 전용 레이아웃 변경 (기존 모바일 대응 그대로)
- 그룹 수동 생성/해제 UI
- 그룹 내 지점 순서 변경

## 기능 상세

### 그룹 감지 로직

```
정규화 키 = item.name.trim().toLowerCase()
같은 키 → 2개 이상: GroupCard
같은 키 → 1개: ItemCard (기존과 동일)
```

그룹 구성은 **필터 적용 전 원본 목록** 기준으로 계산한다.
필터 후 모든 지점이 제외된 그룹은 숨긴다.

### GroupCard

**헤더 (접힌 상태, 기본값):**
```
[카테고리 이모지] [체인명]   [N곳 배지] [▸]
```

**헤더 (펼친 상태):**
```
[카테고리 이모지] [체인명]   [N곳 배지] [▾]
────────────────────────────────
  📍 [주소]        [상태칩] [링크]
  📍 [주소]        [상태칩] [링크]
```

- 기본값: **접힘**
- 헤더 클릭 → 펼치기/접기 토글
- 헤더 자체 클릭은 패널 열지 않음

**지점 수 배지:**
- 필터 없음 또는 전체 표시: `N곳`
- 일부 필터됨: `M/N곳` (M=현재 보이는 수, N=전체)

**카테고리 이모지:**
그룹 내 첫 번째 아이템의 category를 기준으로 표시.

### BranchRow

```
📍  [주소 굵게]                [상태칩들]  [링크버튼]
```

- 주소 없음: `(주소 없음)` 회색 이탤릭
- 행 클릭 → 해당 아이템 패널 열림
- `isActive` 상태: 배경 하이라이트

### 인라인 이름 편집

**ItemCard (단독 아이템):**
- 이름 텍스트 더블클릭 → `<input>` 전환
- Enter / 포커스 아웃 → 저장 후 목록 재구성
- Escape → 취소

**GroupCard 헤더:**
- 이름 텍스트 더블클릭 → `<input>` 전환
- Enter / 포커스 아웃 → 그룹 내 **모든 지점 일괄 rename** (Promise.all 병렬)
- Escape → 취소
- 저장 후 그룹 즉시 재구성

### 정렬

그룹과 단독 아이템을 **혼합 정렬**한다.
그룹의 정렬 키는 그룹 이름 (첫 번째 아이템 name).
정렬 기준별:
- 이름순: 그룹명 알파벳
- 날짜순: 그룹 내 가장 이른 날짜 기준
- 예산순: 그룹 내 최솟값 기준
- 우선순위순: 그룹 내 가장 높은 우선순위 기준

### 지도 연동

- 지도 핀 클릭 (그룹 소속 아이템) → 해당 `GroupCard` 자동 펼침 + BranchRow `isActive`
- 리스트에서 BranchRow 클릭 → 지도 핀 포커스 (기존 동작 유지)

## 비기능 요구사항

- 그룹핑 계산은 `useMemo`로 메모이제이션 (불필요한 재계산 방지)
- 일괄 rename은 낙관적 업데이트 후 병렬 API 호출
