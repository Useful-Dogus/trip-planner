# Feature Specification: Supabase Auth 도입 (회원가입/로그인/재설정)

**Feature Branch**: `082-supabase-auth`
**Created**: 2026-05-19
**Status**: Draft
**Input**: GitHub Issue #107 — [auth] Supabase Auth 도입 (회원가입 / 로그인 / 비밀번호 재설정)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 이메일/비밀번호 로그인 (Priority: P1)

기존 사용자가 이메일과 비밀번호로 trip-planner 에 로그인하여 보호된 페이지(`/list`, `/map`, `/schedule`, `/items`, `/gmaps-import`)에 접근한다.

**Why this priority**: 기존 단일 공유 자격증명 방식에서 다중 사용자 모델로 전환하기 위한 최소 동작 단위. 이 슬라이스만 동작해도 마일스톤 1 의 다음 작업(#108 RLS, #112 대시보드)이 의미를 갖는다.

**Independent Test**: 가입된 사용자가 `/login` 에서 이메일/비밀번호로 로그인 후 `/list` 가 정상 노출되고, 새로고침 후에도 세션이 유지되는지 확인. 로그아웃 후 보호된 URL 직접 접근 시 `/login` 으로 리다이렉트되는지 확인.

**Acceptance Scenarios**:

1. **Given** 유효한 사용자 계정이 존재할 때, **When** 사용자가 올바른 이메일/비밀번호로 로그인하면, **Then** 보호된 페이지로 진입하고 세션이 유지된다.
2. **Given** 로그인된 사용자, **When** 로그아웃하면, **Then** 보호된 페이지 접근 시 `/login` 으로 리다이렉트된다.
3. **Given** 잘못된 비밀번호 입력, **When** 로그인 시도, **Then** 자격증명이 올바르지 않다는 명확한 에러 메시지가 표시되고 세션이 만들어지지 않는다.

---

### User Story 2 - 이메일 회원가입 (Priority: P1)

신규 사용자가 이메일/비밀번호로 계정을 만들고, 가입 확인 메일의 링크를 클릭해 계정을 활성화한 뒤 로그인한다.

**Why this priority**: 다중 사용자 모델의 진입점. 회원가입 없이는 후속 이슈(#108, #111, #112)가 동작할 사용자 자체가 존재하지 않는다.

**Independent Test**: 신규 이메일로 가입 API/페이지를 호출하고 가입 확인 메일을 수신, 확인 링크 클릭 후 동일 자격증명으로 로그인이 가능함을 확인.

**Acceptance Scenarios**:

1. **Given** 미사용 이메일, **When** 사용자가 이메일과 정책에 맞는 비밀번호로 가입하면, **Then** 가입 확인 메일이 발송되고 사용자는 안내 메시지를 본다.
2. **Given** 가입 확인 메일을 받은 사용자, **When** 메일의 확인 링크를 클릭하면, **Then** 계정이 활성화되어 로그인이 가능해진다.
3. **Given** 이미 사용 중인 이메일, **When** 다시 가입 시도, **Then** 중복임을 알리는 에러가 반환되고 새 계정이 만들어지지 않는다.

---

### User Story 3 - 비밀번호 재설정 (Priority: P2)

비밀번호를 잊은 사용자가 자기 이메일로 재설정 메일을 요청하고, 메일 링크를 통해 새 비밀번호를 설정하여 다시 로그인한다.

**Why this priority**: 운영상 필수. 비밀번호 분실 시 복구 수단이 없으면 다중 사용자 운영이 불가하다.

**Independent Test**: 가입된 사용자의 이메일로 재설정 메일을 요청 → 메일 수신 → 링크 클릭 → 새 비밀번호 설정 → 새 비밀번호로 로그인 성공.

**Acceptance Scenarios**:

1. **Given** 가입된 사용자, **When** 자기 이메일로 재설정을 요청하면, **Then** 재설정 메일이 발송된다.
2. **Given** 유효한 재설정 링크, **When** 사용자가 새 비밀번호를 입력해 제출하면, **Then** 비밀번호가 갱신되고 새 자격증명으로 로그인이 가능해진다.
3. **Given** 만료되었거나 이미 사용된 재설정 링크, **When** 사용자가 새 비밀번호 설정을 시도하면, **Then** 명확한 에러 메시지가 표시되고 비밀번호는 변경되지 않는다.

---

### Edge Cases

- 가입 확인 전인 계정으로 로그인 시도할 때의 처리(Supabase 기본 동작: 확인 전 로그인 불가) — 사용자에게 메일 확인을 안내한다.
- 동일 이메일로 가입 확인 메일 재전송 요청은 본 이슈 범위 밖(후속 #111 UI 에서 처리).
- 로그아웃 직후 다른 탭에서의 세션 만료 처리 — 다음 페이지 진입 시 `/login` 리다이렉트로 자연 해소.
- 보호된 API 가 만료/누락된 세션을 받았을 때 — HTTP 401 응답.
- 보호된 페이지에 미인증 접근 — `/login` 으로 리다이렉트.
- 잘못된 형식의 이메일/짧은 비밀번호 등 입력 오류 — 사용자가 이해할 수 있는 메시지로 반환.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 이메일과 비밀번호로 신규 사용자 계정을 생성할 수 있어야 한다.
- **FR-002**: 시스템은 신규 가입 시 사용자 이메일로 가입 확인 메일을 발송해야 한다.
- **FR-003**: 가입 확인 메일의 링크를 클릭한 사용자는 계정이 활성화되어 로그인할 수 있어야 한다.
- **FR-004**: 시스템은 이메일과 비밀번호로 인증된 사용자에게만 보호된 페이지(`/list`, `/map`, `/schedule`, `/items`, `/gmaps-import`)와 보호된 API(`/api/items`, `/api/geocode`, `/api/gmaps`) 접근을 허용해야 한다.
- **FR-005**: 사용자가 로그인하면 세션이 안전한 방식(HttpOnly 쿠키)으로 유지되어, 브라우저 새로고침 후에도 인증 상태가 유지되어야 한다.
- **FR-006**: 사용자는 로그아웃 액션으로 세션을 즉시 종료할 수 있어야 한다.
- **FR-007**: 시스템은 사용자의 이메일로 비밀번호 재설정 메일 요청을 받아 재설정 링크를 발송할 수 있어야 한다.
- **FR-008**: 유효한 재설정 링크를 통해 들어온 사용자는 새 비밀번호를 설정할 수 있고, 변경 후 새 비밀번호로 로그인할 수 있어야 한다.
- **FR-009**: 만료되었거나 이미 사용된 재설정 링크는 거부되어야 한다.
- **FR-010**: 잘못된 자격증명 또는 입력 오류는 명확한 한국어 에러 메시지로 사용자에게 전달되어야 한다.
- **FR-011**: 기존에 사용되던 단일 공유 자격증명(`AUTH_ID`/`AUTH_PASSWORD`) 및 자체 서명 비밀(`JWT_SECRET`) 의존성은 더 이상 사용되지 않고 시스템에서 제거되어야 한다.
- **FR-012**: 인증 검증을 수행하는 코드 경로는 단일 어댑터 모듈(`lib/auth.ts`)에 한정되어야 하며, 다른 모듈은 외부 인증 벤더의 API 를 직접 호출하지 않아야 한다 (벤더 이전 시 교체 비용 최소화 목적).
- **FR-013**: 라우트 보호 미들웨어의 외부 호출부는 회귀 없이 동작해야 한다 (보호 라우트 목록·리다이렉트 정책 동일 유지).

### Key Entities

- **User Account**: 시스템 내 한 명의 사용자. 식별자는 이메일과 시스템이 부여한 안정적 사용자 ID. 활성/미활성(가입 확인 전) 상태를 가진다.
- **Session**: 한 사용자가 로그인 상태로 유지되는 단위. 쿠키 기반으로 브라우저에 유지되며 만료 시간을 가진다.
- **Password Reset Token**: 비밀번호 재설정 메일 링크에 포함되는 일회성 자격. 만료/사용 상태를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 가입 시작부터 가입 확인 메일 수신까지 60 초 이내에 도달한다 (메일 시스템 정상 가정).
- **SC-002**: 가입된 사용자가 이메일/비밀번호 입력부터 보호된 페이지 첫 화면 로딩까지 5 초 이내에 도달한다.
- **SC-003**: 비밀번호 재설정 요청부터 새 비밀번호로 로그인 성공까지 3 분 이내에 완료할 수 있다 (메일 수신 대기 포함, 메일 시스템 정상 가정).
- **SC-004**: 보호된 페이지 또는 보호된 API 에 인증 없이 접근하는 100% 의 요청이 차단된다 (페이지는 `/login` 리다이렉트, API 는 401).
- **SC-005**: 기존 회귀 시나리오(로그인 후 보호된 페이지 접근, 로그아웃 후 차단, 잘못된 자격증명 거부) 100% 가 동일하게 동작한다.
- **SC-006**: 인증 관련 외부 의존 호출 지점이 코드베이스에서 1 개 모듈로 격리되어 있다 (벤더 이전 시 1 개 파일 수정으로 교체 가능한 구조 유지).

## Out of Scope

- OAuth(Google/GitHub 등) 소셜 로그인 — 후속.
- 권한/멤버십 모델·RLS 정책 — #108.
- 회원가입/재설정의 완성된 풀세트 UI(전용 폼, 안내 페이지, 디자인 정합) — #111. 본 이슈에서는 동작 가능한 최소 UI/엔드포인트만 제공.
- 커스텀 발송 메일 서버·맞춤 템플릿 디자인 — Supabase 기본 메일 사용.
- 사용자 프로필·아바타·표시명 등 부가 정보 — 후속.

## Assumptions

- Supabase 프로젝트가 이미 존재하고 검증 시점에 깨어 있다 (paused 아님).
- Supabase Authentication > Providers > Email 이 활성화되어 있다 (기본값 ON).
- Supabase Authentication > URL Configuration 에 로컬(`http://localhost:3000`) 및 운영 도메인이 등록되어 있다 — 그렇지 않으면 메일 링크 리다이렉트가 동작하지 않는다.
- 메일 발송은 Supabase 기본 인프라를 사용하며, 본 이슈는 발송 성공률·도달률을 제품 목표로 두지 않는다.
- 비밀번호 정책은 Supabase 기본값을 따른다 (필요 시 후속 이슈에서 강화).
- 기존 단일 공유 자격증명으로 사용 중이던 사용자는 본 변경 적용 시점에 새 계정으로 가입을 다시 수행한다 (개인 dev 단계로 마이그레이션 부담 없음).

## Dependencies

- 마일스톤 1 이슈 #108(RLS + `trip_members`) 은 본 이슈가 만들어내는 사용자 ID 를 입력으로 사용한다.
- 마일스톤 1 이슈 #111(인증 UI) 은 본 이슈의 엔드포인트/어댑터를 사용한다.
