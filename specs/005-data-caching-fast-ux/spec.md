# Feature Specification: 데이터 캐싱 및 빠른 UX 전략

**Feature Branch**: `005-data-caching-fast-ux`
**Created**: 2026-03-27
**Status**: Draft
**Input**: User description: "데이터 캐싱 #51 하고싶어. 볼때마다 로딩되지는 않는지 궁금함. 빠르게 조회하고 수정하고 확인할수 있는게 중요해. 캐싱 외에도 데이터를 빠르다는 느낌을 받게하는 다양한 UX 전략을 종합 검토하고 스펙을 작성해줘."

---

## 배경

현재 앱은 페이지 전환(리서치 탭 ↔ 일정 탭 ↔ 지도 탭)을 할 때마다 서버에서 전체 아이템 목록을 새로 불러온다. 탭을 오가며 빠르게 확인·수정하는 워크플로우에서 매번 로딩 스피너가 보이는 것은 불필요한 마찰이다. 아이템 수정 후에도 저장이 완료될 때까지 UI가 멈춰 있어 "느리다"는 인상을 준다.

이 기능은 **캐싱**으로 중복 네트워크 요청을 줄이고, **낙관적 업데이트**로 수정 반응을 즉각적으로 만들며, 그 외 **지각된 성능(perceived performance)** UX 전략을 종합 적용하여 앱 전반이 빠르게 느껴지도록 한다.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 탭 전환 시 즉시 표시 (Priority: P1)

사용자가 리서치 탭에서 지도 탭으로, 다시 일정 탭으로 이동할 때 이전에 불러온 데이터가 즉시 표시되고 로딩 스켈레톤이 다시 나타나지 않는다.

**Why this priority**: 가장 빈번한 사용 패턴이며, 매번 보이는 스켈레톤이 가장 큰 체감 불편 요소다.

**Independent Test**: 앱을 열어 아이템 목록이 로드된 뒤, 탭을 연속으로 3번 전환한다. 첫 로드 이후에는 스켈레톤이 보이지 않고 이전 데이터가 즉시 렌더링되면 통과.

**Acceptance Scenarios**:

1. **Given** 리서치 탭이 이미 로드된 상태에서, **When** 사용자가 일정 탭을 클릭하면, **Then** 일정 탭이 스켈레톤 없이 즉시 아이템 목록을 보여준다.
2. **Given** 탭을 여러 번 오간 상태에서, **When** 네트워크가 정상인 경우, **Then** 데이터는 백그라운드에서 최신화되고 화면은 이전 데이터를 유지한다.
3. **Given** 캐시된 데이터가 있는 상태에서, **When** 앱을 새로고침(페이지 리로드)하면, **Then** 캐시된 데이터를 즉시 표시하고 백그라운드에서 최신 데이터로 교체한다.

---

### User Story 2 - 아이템 수정 즉시 반영 (Priority: P1)

사용자가 아이템 이름·상태·날짜 등을 수정하고 저장을 누르면, 서버 응답을 기다리지 않고 즉시 목록에 반영된다. 저장 실패 시에는 되돌린다.

**Why this priority**: 수정 → 확인 반복이 핵심 워크플로우이며, 저장 후 "빠르게 확인할 수 있는 것"이 핵심 요구사항이다.

**Independent Test**: 아이템 이름을 수정하고 저장을 누른다. 저장 버튼을 누른 직후 패널이 닫히고 목록에 변경된 이름이 즉시 표시되면 통과.

**Acceptance Scenarios**:

1. **Given** 사용자가 아이템 이름을 수정하고 저장을 누르면, **When** 저장 버튼 클릭 즉시, **Then** 패널이 닫히고 목록의 해당 아이템에 새 이름이 반영된다 (서버 응답 대기 없음).
2. **Given** 낙관적으로 반영된 수정이 서버에서 실패하면, **When** 서버 오류 응답 수신 시, **Then** 목록이 이전 값으로 되돌아가고 사용자에게 실패 메시지가 표시된다.
3. **Given** 아이템 삭제 버튼을 누르면, **When** 즉시, **Then** 목록에서 해당 아이템이 사라지고 백그라운드에서 서버 삭제 요청이 전송된다.

---

### User Story 3 - 신규 아이템 추가 즉시 목록 반영 (Priority: P2)

사용자가 새 아이템을 추가하면 서버 저장 완료를 기다리지 않고 목록에 즉시 나타난다.

**Why this priority**: 추가 → 확인 흐름도 빠른 피드백이 중요하지만, 수정보다는 덜 빈번하다.

**Independent Test**: 새 아이템 추가 후 패널을 닫았을 때 목록에 즉시 나타나면 통과.

**Acceptance Scenarios**:

1. **Given** 사용자가 새 아이템 폼을 작성하고 저장을 누르면, **When** 저장 버튼 클릭 즉시, **Then** 목록에 임시 ID로 새 아이템이 표시된다.
2. **Given** 서버가 아이템 생성에 성공하면, **When** 응답 수신 시, **Then** 임시 ID가 실제 서버 ID로 교체되고 UI 변화는 없다.
3. **Given** 서버 생성이 실패하면, **When** 오류 응답 수신 시, **Then** 목록에서 임시 아이템이 제거되고 실패 알림이 표시된다.

---

### User Story 4 - 빠른 상태 토글 (Priority: P2)

아이템 카드의 상태(검토중/확정/탈락 등)를 변경할 때 즉각 반영된다. 매번 저장 패널을 열 필요 없이 빠르게 상태를 바꿀 수 있다.

**Why this priority**: 반복적인 상태 변경이 잦은 계획 단계에서 큰 효율 차이를 만든다.

**Independent Test**: 아이템 카드에서 상태를 변경하면 카드 뱃지가 즉시 바뀌면 통과.

**Acceptance Scenarios**:

1. **Given** 아이템 카드에서 상태 변경 버튼을 클릭하면, **When** 즉시, **Then** 카드의 상태 뱃지가 새 상태로 바뀐다.
2. **Given** 상태 변경 서버 요청이 실패하면, **When** 오류 응답 시, **Then** 뱃지가 원래 상태로 복원된다.

---

### User Story 5 - 오프라인/네트워크 불안정 상황 대응 (Priority: P3)

네트워크가 불안정하거나 일시적으로 끊겨도 마지막 캐시 데이터를 표시하고, 재연결 시 자동으로 최신화된다.

**Why this priority**: 이동 중 여행 계획을 짜는 시나리오에서 유용하지만 핵심 기능은 아니다.

**Independent Test**: 앱 로드 후 네트워크를 끊고 탭을 전환해도 캐시 데이터가 표시되면 통과.

**Acceptance Scenarios**:

1. **Given** 앱이 한 번 로드된 상태에서, **When** 네트워크가 끊기고 탭을 전환하면, **Then** 마지막으로 캐시된 데이터를 보여준다.
2. **Given** 네트워크가 재연결되면, **When** 자동 감지 시, **Then** 백그라운드에서 데이터를 갱신하고 화면을 최신화한다.

---

### Edge Cases

- 두 개의 탭/기기에서 동시에 같은 아이템을 수정하면? → 나중에 저장한 쪽이 우선 적용되며, 이전 캐시는 덮어쓴다.
- 낙관적 업데이트 중 다시 동일 아이템을 수정하면? → 이미 진행 중인 요청이 있어도 최신 수정값으로 덮어쓴다.
- 캐시가 오래됐을 때(장시간 탭 방치 후 복귀)? → 포커스 복귀 시 백그라운드 자동 갱신으로 최신화.
- 서버 오류로 낙관적 업데이트 롤백 중 사용자가 또 수정하면? → 롤백을 완료하고 새 수정을 적용한다.
- 캐시가 없는 최초 접속 상태? → 기존과 동일하게 로딩 스켈레톤을 보인다.

## Requirements *(mandatory)*

### Functional Requirements

**캐싱 & 데이터 동기화**

- **FR-001**: 시스템은 최초 로드 이후 동일 세션 내 탭 전환 시 서버 재요청 없이 메모리에 캐시된 데이터를 즉시 표시해야 한다.
- **FR-002**: 시스템은 캐시된 데이터를 표시하는 동시에 백그라운드에서 서버 최신 데이터를 조회(stale-while-revalidate)해야 한다.
- **FR-003**: 시스템은 앱이 백그라운드에서 포그라운드로 복귀할 때(탭/윈도우 포커스 획득 시) 자동으로 데이터를 갱신해야 한다. 주기적 백그라운드 폴링은 제공하지 않는다.
- **FR-004**: 시스템은 페이지 새로고침 시에도 영속 저장소에 캐시된 데이터를 즉시 표시하고 백그라운드에서 갱신해야 한다.
- **FR-005**: 캐시는 마지막 성공 응답 기준으로 유지되며, 서버 갱신 성공 시 자동 갱신된다.

**낙관적 업데이트**

- **FR-006**: 아이템 수정(저장) 시 서버 응답 전에 즉시 UI에 변경사항을 반영해야 한다.
- **FR-007**: 아이템 삭제 시 서버 응답 전에 즉시 목록에서 제거해야 한다.
- **FR-008**: 아이템 추가 시 서버 응답 전에 즉시 목록에 임시 항목을 표시해야 한다.
- **FR-009**: 서버 요청이 실패하면 낙관적으로 변경된 UI를 이전 상태로 자동 롤백해야 한다.
- **FR-010**: 롤백 발생 시 자동 소멸 토스트(약 4초)로 실패 알림을 표시하며, 토스트에는 해당 작업을 즉시 재시도할 수 있는 버튼을 포함해야 한다.

**빠른 상태 변경 UX**

- **FR-011**: 아이템 카드의 상태 뱃지를 클릭하면 전체 상태 목록(검토중/보류/대기중/확정/탈락)을 보여주는 드롭다운/팝오버가 표시되며, 선택 즉시 상태가 변경된다 (패널을 열지 않아도 상태 변경 가능).
- **FR-012**: 상태 변경은 낙관적으로 즉시 반영되어야 한다 (FR-009, FR-010 동일 적용).

**오프라인 대응**

- **FR-013**: 네트워크가 없는 상태에서도 캐시된 데이터를 표시하고 읽기 전용으로 사용할 수 있어야 한다.
- **FR-014**: 네트워크 재연결 감지 시 자동으로 최신 데이터를 가져와 화면을 업데이트해야 한다.

### Key Entities

- **아이템 캐시**: 전체 TripItem 목록의 최신 스냅샷. 메모리(세션 내)와 영속 스토리지(새로고침 후에도 유지) 두 계층으로 구성.
- **낙관적 상태**: 서버 응답 전 UI에 반영된 임시 상태. 서버 성공 시 확정, 실패 시 이전 상태로 롤백.
- **동기화 상태**: 캐시의 신선도 정보 (마지막 갱신 시각, 갱신 중 여부, 오류 여부).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 최초 로드 이후 탭 전환 시 아이템 목록이 로딩 스켈레톤 없이 즉시 표시된다.
- **SC-002**: 아이템 저장·삭제 버튼 클릭 후 0.1초 이내에 목록에 변경사항이 반영된다 (서버 응답 무관).
- **SC-003**: 페이지 새로고침 후에도 서버 응답 전에 이전 데이터가 즉시 표시된다 (빈 화면 없음).
- **SC-004**: 포그라운드 복귀(탭 클릭, 창 전환) 시 30초 이상 방치된 경우 자동으로 백그라운드 갱신이 시작된다.
- **SC-005**: 서버 요청 실패 시 3초 이내에 UI가 이전 상태로 복원되고 실패 알림이 표시된다.
- **SC-006**: 네트워크 오프라인 상태에서 캐시 데이터로 정상 탐색이 가능하며 오프라인 표시가 나타난다.
- **SC-007**: 아이템 카드에서 패널을 열지 않고 상태를 직접 변경할 수 있다.

## Clarifications

### Session 2026-03-27

- Q: 아이템 카드에서 상태 토글 시 인터랙션 방식은? → A: 뱃지 클릭 시 상태 목록 드롭다운/팝오버 표시 후 선택
- Q: 낙관적 업데이트 실패 시 알림 형태는? → A: 자동 소멸 토스트(약 4초) + 재시도 버튼 포함
- Q: 백그라운드 자동 갱신 주기는? → A: 포커스 복귀 시에만 갱신, 주기적 폴링 없음

## Assumptions

- 단일 사용자 앱이므로 동시 편집 충돌은 예외적 시나리오로 간주하며 "마지막 저장 승리" 전략을 적용한다.
- 전체 아이템 목록이 수백 건 이내로 유지된다고 가정하므로, 전체 목록을 한 번에 캐시하는 방식이 적합하다.
- 영속 캐시의 최대 보존 기간은 24시간으로 설정한다. 24시간 초과 시 만료 처리하고 서버에서 새로 조회한다.
- 오프라인 중 쓰기 작업(추가·수정·삭제)은 이 기능 범위에서 제외한다. 오프라인 시 읽기만 지원한다.
- 상태 토글 UI는 기존 카드 디자인을 최소한으로 수정하는 방향으로 구현한다.
