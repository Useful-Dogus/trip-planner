# Feature Specification: 일정 종료 시간 및 상세 일정 표시 개선

**Feature Branch**: `008-item-end-time`
**Created**: 2026-04-06
**Status**: Draft
**Input**: User description: "일정 종료 시간을 추가하고, 디테일 모달에서 시작 날짜 / 시작 시간 / 종료 날짜 / 종료 시간을 각각 다른 행에 표시한다."

## Context

현재 `TripItem`은 `date`와 `time_start`를 통해 단일 시작 시점만 표현한다. 데이터 계층과 스키마에는 `time_end` 필드가 일부 반영되어 있으나, 생성/편집 폼, API 검증, 상세 표시 UI는 이를 완전히 지원하지 않는다.

호텔처럼 체류형 일정이나 종료 시각이 중요한 일정은 종료 시각을 함께 입력해야 실제 이동/체류 흐름을 정확히 표현할 수 있다. 또한 상세 패널의 일정 영역은 `날짜`와 `시작 시간`만 보여주고 있어 기간성 일정의 정보를 충분히 전달하지 못한다.

이번 변경은 기존 일정 모델을 분리하지 않고, 옵셔널 종료 시각을 추가해 단일 시점 일정과 기간형 일정을 모두 표현 가능하게 만드는 범위다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 종료 시간이 있는 일정을 저장하기 (Priority: P1)

사용자는 항목 생성/편집 시 종료 시간을 입력해 저장할 수 있다.

**Why this priority**: 종료 시간 저장이 되지 않으면 기간형 일정 표현 자체가 불가능하다.

**Independent Test**: 생성 폼과 편집 폼에서 종료 시간을 입력한 뒤 저장하고 다시 열었을 때 값이 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 새 항목 폼을 연 상태, **When** 종료 시간을 입력하고 저장하면, **Then** 항목 데이터에 `time_end`가 저장된다.
2. **Given** 기존 항목에 종료 시간이 저장된 상태, **When** 편집 폼을 열면, **Then** 종료 시간 입력란에 기존 값이 채워져 있다.
3. **Given** 사용자가 종료 시간을 비운 상태로 저장하면, **When** 항목이 저장되면, **Then** `time_end`는 선택값으로 유지되고 저장이 실패하지 않는다.

---

### User Story 2 - 상세 화면에서 시작/종료 정보를 분리해 읽기 (Priority: P1)

사용자는 디테일 모달과 상세 페이지에서 시작 날짜, 시작 시간, 종료 날짜, 종료 시간을 각각 독립된 행으로 읽을 수 있다.

**Why this priority**: 이번 요청의 핵심 UI 개선이며, 일정의 구조를 한눈에 읽을 수 있어야 한다.

**Independent Test**: 시작/종료 정보가 모두 있는 항목을 열어 4개 행이 각각 별도 라벨로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 항목에 `date`, `time_start`, `time_end`가 있는 상태, **When** 상세 패널을 열면, **Then** 일정 섹션에 `시작 날짜`, `시작 시간`, `종료 날짜`, `종료 시간`이 각각 다른 행으로 표시된다.
2. **Given** 종료 시간이 없는 항목, **When** 상세 패널을 열면, **Then** `종료 시간` 행은 숨겨지고 시작 정보만 표시된다.
3. **Given** 항목 상세 페이지를 여는 상태, **When** 일정 섹션을 보면, **Then** 상세 패널과 동일한 라벨 구조로 표시된다.

## Edge Cases

- 종료 시간만 있고 시작 날짜가 없는 비정상 입력은 API에서 허용하지 않거나, 시작 날짜/시간과 동일한 문맥에서만 사용되도록 검토해야 한다.
- 종료 시간이 시작 시간보다 빠른 경우(자정 넘김)는 현재 모델상 날짜 분리가 없으므로 단순 시각 저장만 허용하고, 날짜 간 해석은 추후 과제로 남긴다.
- 기존 데이터에는 `time_end`가 없을 수 있으므로 UI는 값이 없을 때 자연스럽게 시작 정보만 보여줘야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 `TripItem` 생성 및 수정 시 `time_end`를 옵셔널 필드로 저장할 수 있어야 한다.
- **FR-002**: 시스템은 `time_end`가 제공되면 `HH:MM` 형식으로 검증해야 한다.
- **FR-003**: 시스템은 새 항목 생성 폼과 기존 항목 편집 폼에서 `종료 시간` 입력란을 제공해야 한다.
- **FR-004**: 시스템은 상세 패널 일정 섹션에서 `시작 날짜`, `시작 시간`, `종료 날짜`, `종료 시간`을 각각 독립 행으로 표시해야 한다.
- **FR-005**: 시스템은 상세 페이지 일정 섹션에서도 상세 패널과 동일한 라벨 구조를 사용해야 한다.
- **FR-006**: 시스템은 종료 날짜 전용 필드를 새로 추가하지 않고, 현재 일정의 종료 날짜는 시작 날짜와 동일한 날짜 값으로 표시해야 한다.
- **FR-007**: 시스템은 종료 시간이 없는 기존 항목도 회귀 없이 표시하고 저장할 수 있어야 한다.

### Key Entities

- **TripItem**: 여행 항목. `date`, `time_start`에 더해 옵셔널 `time_end`를 가진다.
- **ScheduleDetailRows**: 상세 패널과 상세 페이지에서 일정 정보를 `시작 날짜 / 시작 시간 / 종료 날짜 / 종료 시간` 구조로 표시하는 UI 표현 규칙.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 생성/편집 폼에서 종료 시간을 입력해 저장하고 다시 열었을 때 동일 값을 확인할 수 있다.
- **SC-002**: 상세 패널과 상세 페이지에서 일정 정보가 요청한 4개 라벨 구조로 일관되게 표시된다.
- **SC-003**: 종료 시간이 없는 기존 항목도 생성/수정/상세 보기 흐름에서 오류 없이 동작한다.
