# Feature Specification: Trip Pulse — 룰 기반 여행 상태 내레이션

**Feature Branch**: `328-trip-pulse`  
**Created**: 2026-06-25  
**Status**: Draft  
**Input**: GitHub Issue [#328](https://github.com/Useful-Dogus/trip-planner/issues/328) + 사용자 방향 결정: 단계 강제/작업 큐가 아니라 여행이 선명해지는 느낌을 주는 룰 기반 1단계 요약

## Background

Waypost는 여행 후보를 모으고, 살펴보고, 일정으로 이어가는 앱이다. 하지만 현재 화면들은 목록·지도·일정 기능을 각각 제공할 뿐, 사용자가 모은 장소들이 어떤 여행으로 자라고 있는지 부드럽게 비춰주지 않는다.

초기 논의에서 "현재 단계 표시" 또는 "다음 작업 큐" 방식이 검토되었지만, 이는 사용자의 주도권을 빼앗거나 여행 계획을 업무처럼 느끼게 만들 위험이 있다. 사용자는 자기 상황에 맞게 목록, 지도, 일정을 오가며 계획한다. 따라서 이 기능은 단계를 판정하거나 과제를 부여하지 않고, 현재 여행 데이터에서 읽을 수 있는 작은 상태를 즐거운 문장으로 보여준다.

이번 범위는 룰 기반 1단계에 한정한다. LLM, 개인화 학습, 저장된 진행 상태, 자동 추천, 자동 일정 생성은 다루지 않는다.

## Clarifications

- **표시 위치**: 목록·일정 화면에서는 기존 trip 헤더 아래에 표시한다. 지도 화면에서는 지도 위를 덮는 큰 오버레이가 아니라 기존 지도 사이드 패널 상단에 표시한다.
- **액션 포함 여부**: v1은 선택적 이동 액션을 허용하되 화면별 최대 1개만 표시한다. 문장이 중심이고 액션은 작은 보조 진입점이다.
- **문장 우선순위**: "가장 여행이 구체화된 사실"을 우선한다. 기본 우선순위는 `일정에 들어간 장소` > `확정한 장소` > `가고 싶은 곳` > `지도에 표시 가능한 곳` > `전체 후보` > `빈 상태`이다.
- **숨김 설정**: v1에는 닫기·숨기기·사용자 설정을 두지 않는다. 표시 밀도와 톤을 낮춰 항상 보여도 부담 없는 컴포넌트로 만든다.
- **보조 문장**: 스펙은 보조 문장 1개를 허용하지만 v1 구현은 기본 문장 1개를 우선한다. 보조 문장은 구현 중 실제 화면 밀도가 과하지 않을 때만 사용한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 모은 장소가 여행으로 자라는 느낌을 본다 (Priority: P1)

여행자는 목록 화면에서 지금까지 모은 후보와 확정 장소가 어떤 상태인지 짧은 문장으로 본다. 문장은 "해야 할 일"이 아니라 "가고 싶은 곳이 모이고 있다"는 감각을 준다.

**Why this priority**: 목록은 후보 수집과 취향 정리의 주 표면이다. 이곳에서 여행의 윤곽이 보이지 않으면 앱은 단순 테이블처럼 느껴진다.

**Independent Test**: 항목 구성이 다른 trip 3개(빈 trip, 후보만 있는 trip, 확정 항목이 있는 trip)를 만들어 목록 화면을 열고, 각 상태에 맞는 문장이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** trip에 항목이 없을 때, **When** 사용자가 목록 화면을 열면, **Then** "아직 첫 장소를 기다리고 있어요"처럼 빈 상태를 부드럽게 설명하는 문장이 보인다.
2. **Given** `가고 싶음` 항목이 3개 이상 있을 때, **When** 사용자가 목록 화면을 열면, **Then** "가고 싶은 곳이 N곳 모였어요"처럼 기대감을 주는 문장이 보인다.
3. **Given** `확정` 항목이 1개 이상 있을 때, **When** 사용자가 목록 화면을 열면, **Then** "N곳이 이번 여행에 들어갔어요"처럼 여행이 구체화되고 있음을 보여준다.
4. **Given** 여러 문장 후보가 동시에 참일 때, **When** 화면이 렌더링되면, **Then** 가장 구체적이고 긍정적인 문장 1개가 우선 표시된다.

---

### User Story 2 - 지도에서 여행의 공간감을 느낀다 (Priority: P2)

여행자는 지도 화면에서 현재 장소들이 지도 위에 얼마나 드러났는지, 어떤 후보가 공간적으로 보이는지 짧게 읽는다.

**Why this priority**: 지도는 계획의 공간감을 만드는 화면이다. 장소 수나 확정 수를 업무 지표처럼 보여주기보다, "지도에 여행이 펼쳐지고 있다"는 느낌을 주는 문장이 필요하다.

**Independent Test**: 좌표가 있는 항목과 없는 항목이 섞인 trip으로 지도 화면을 열고, 지도에 표시 가능한 장소 수 중심의 문장이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 좌표가 있는 항목이 1개 이상 있을 때, **When** 사용자가 지도 화면을 열면, **Then** "지도에 N곳을 펼쳐봤어요"처럼 지도 표면에 맞는 문장이 보인다.
2. **Given** `확정`이면서 좌표가 있는 항목이 1개 이상 있을 때, **When** 사용자가 지도 화면을 열면, **Then** "이번 여행에 들어간 N곳을 지도에서 보고 있어요"처럼 확정 장소의 공간감을 보여준다.
3. **Given** 좌표 없는 항목이 있어도, **When** 문장이 표시되면, **Then** "위치 없음 N개 처리 필요" 같은 업무형 표현은 쓰지 않는다.

---

### User Story 3 - 일정에서 하루들이 채워지는 느낌을 본다 (Priority: P2)

여행자는 일정 화면에서 확정 장소가 날짜와 시간으로 얼마나 여행의 하루를 채우고 있는지 짧은 문장으로 본다.

**Why this priority**: 일정은 계획이 실제 여행일로 바뀌는 표면이다. "미완료"보다 "하루가 채워지고 있다"는 감각을 우선해야 한다.

**Independent Test**: 날짜 없는 확정 항목, 날짜만 있는 항목, 시간까지 있는 항목이 섞인 trip으로 일정 화면을 열고, 날짜가 있는 확정 항목 중심의 문장이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 날짜가 있는 `확정` 항목이 있을 때, **When** 사용자가 일정 화면을 열면, **Then** "N곳이 일정에 들어갔어요"처럼 계획이 구체화된 상태를 보여준다.
2. **Given** 서로 다른 날짜에 확정 항목이 분포할 때, **When** 사용자가 일정 화면을 열면, **Then** "M일의 여행이 채워지고 있어요"처럼 날짜 단위 윤곽을 보여줄 수 있다.
3. **Given** 날짜 없는 확정 항목이 있어도, **When** 문장이 표시되면, **Then** "미배정 N개" 같은 작업 관리 문구 대신 "아직 날짜를 정하지 않은 곳도 있어요"처럼 부드럽게 표현한다.

---

### User Story 4 - 사용자는 원하면 이어서 탐색한다 (Priority: P3)

여행자는 요약 문장 옆의 작은 액션을 통해 자연스럽게 다른 표면으로 이동할 수 있다. 액션은 명령이 아니라 탐색의 초대여야 한다.

**Why this priority**: #328의 원래 문제는 기능들이 이어지지 않는다는 점이다. 다만 사용자를 다음 단계로 몰아가면 제품 철학과 맞지 않으므로, 작은 선택지로만 제공한다.

**Independent Test**: 각 화면의 액션을 눌러 기존 URL 상태를 깨지 않고 대상 화면으로 이동하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 목록 화면에 지도 액션이 보일 때, **When** 사용자가 액션을 클릭하면, **Then** 같은 trip의 지도 화면으로 이동한다.
2. **Given** 지도 화면에 일정 액션이 보일 때, **When** 사용자가 액션을 클릭하면, **Then** 같은 trip의 일정 화면으로 이동한다.
3. **Given** 일정 화면에 목록 액션이 보일 때, **When** 사용자가 액션을 클릭하면, **Then** 같은 trip의 목록 화면으로 이동한다.
4. **Given** 액션이 현재 화면과 같은 목적지를 가리킬 필요가 없을 때, **When** 문장이 렌더링되면, **Then** 액션은 생략 가능하다.

### Edge Cases

- **빈 trip**: 후보가 없을 때도 실패하거나 과장하지 않고 첫 장소를 기다리는 문장을 표시한다.
- **전부 제외된 trip**: 제외 항목만 있으면 "여행이 완성됐다"처럼 오해되는 문장을 표시하지 않는다.
- **좌표 없는 항목**: 지도 화면에서 좌표 없는 항목을 오류처럼 다루지 않는다. 지도 문장은 지도에 실제 표시 가능한 항목을 기준으로 한다.
- **날짜 없는 확정 항목**: 일정 화면에서 부드럽게 언급할 수 있으나 작업 큐처럼 강조하지 않는다.
- **viewer 권한**: viewer도 문장을 볼 수 있다. 단, 쓰기 액션이 필요한 버튼은 표시하지 않거나 읽기 전용 이동만 제공한다.
- **모바일 좁은 화면**: 문장은 1-2줄 안에서 자연스럽게 줄바꿈되어야 하며, 버튼 텍스트가 잘리지 않아야 한다.
- **데이터 로딩 중**: 로딩 중에는 skeleton 또는 미표시 상태를 사용하고, 0개 상태 문장을 먼저 보여주지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 현재 trip의 `items` 배열만 사용해 화면별 여행 상태 문장을 생성해야 한다.
- **FR-002**: 시스템은 저장된 진행 단계, 사용자별 학습 데이터, 외부 API, LLM 호출 없이 문장을 생성해야 한다.
- **FR-003**: 시스템은 목록, 지도, 일정 화면 각각에 맞는 문장 후보 집합을 가져야 한다.
- **FR-004**: 시스템은 한 화면에서 기본 문장 1개를 표시해야 하며, 보조 문장은 최대 1개까지만 허용한다. v1 구현은 기본 문장 1개를 우선한다.
- **FR-005**: 문장 선택은 결정적이어야 한다. 같은 `items`와 같은 화면에서는 동일한 문장이 표시되어야 한다.
- **FR-006**: 문장 우선순위는 "가장 여행이 구체화된 사실"을 우선해야 한다. 기본 우선순위는 `일정에 들어간 장소` > `확정한 장소` > `가고 싶은 곳` > `지도에 표시 가능한 곳` > `전체 후보` > `빈 상태`이다.
- **FR-007**: 문장은 작업 관리형 표현을 피해야 한다. 금지 표현 예: "미완료", "처리 필요", "다음 단계", "완성도", "작업", "태스크".
- **FR-008**: 문장은 여행 계획의 즐거움과 윤곽을 드러내는 표현을 사용해야 한다. 허용 표현 예: "모였어요", "들어갔어요", "펼쳐봤어요", "채워지고 있어요", "아직 정하지 않은 곳".
- **FR-009**: 목록 화면 문장은 `trip_priority` 분포를 활용할 수 있어야 한다. 최소한 전체 항목 수, `가고 싶음`, `확정`, `검토 필요`, `제외` 수를 구분한다.
- **FR-010**: 지도 화면 문장은 좌표가 있는 항목 수와 좌표가 있는 확정 항목 수를 활용할 수 있어야 한다.
- **FR-011**: 일정 화면 문장은 날짜가 있는 확정 항목 수, 확정 항목이 분포한 날짜 수, 날짜 없는 확정 항목 수를 활용할 수 있어야 한다.
- **FR-012**: 시스템은 각 화면에서 선택적으로 1개의 이동 액션을 제공할 수 있어야 한다. 액션은 같은 trip의 기존 화면으로만 이동하며, 화면별 최대 1개만 표시한다.
- **FR-013**: 액션 문구는 명령형 작업 지시가 아니라 탐색형 문구여야 한다. 예: "지도에서 보기", "일정에서 보기", "목록으로 돌아가기".
- **FR-014**: 기능은 기존 검색, 필터, 정렬, 선택 패널, URL search params 동작을 변경하지 않아야 한다.
- **FR-015**: 기능은 trip item 데이터의 생성, 수정, 삭제, 우선순위 변경, 날짜 변경을 수행하지 않아야 한다.
- **FR-016**: 문장 생성 로직은 UI와 분리되어 단위 테스트 가능한 순수 함수로 둘 수 있어야 한다.
- **FR-017**: 컴포넌트는 모바일과 데스크탑 양쪽에서 표시되어야 하며, 주요 콘텐츠를 밀어내거나 지도 조작을 방해하지 않아야 한다. 닫기·숨기기·사용자 설정은 v1 범위에 포함하지 않는다.
- **FR-018**: 지도 화면에서는 지도 위를 가리는 큰 카드가 아니라 기존 지도 사이드 패널 상단에 표시해야 한다.
- **FR-019**: 로딩 중에는 실제 항목 수를 알 수 없으므로 빈 상태 문장을 표시하지 않아야 한다.

### Key Entities

- **Trip Pulse Summary**: 현재 화면에 표시할 기본 문장, 선택적 보조 문장, 선택적 이동 액션을 포함하는 표시용 값. 영구 저장하지 않는다.
- **Trip Pulse Metrics**: `items`에서 계산한 집계 값. 전체 항목 수, 우선순위별 수, 좌표 보유 수, 날짜 보유 확정 항목 수 등을 포함한다. 영구 저장하지 않는다.
- **Trip Pulse Surface**: 문장이 표시되는 화면 종류. 값은 `list`, `map`, `schedule`로 제한한다.

## Copy Guidelines

- 사용자를 평가하지 않는다.
- 진행률이나 점수처럼 보이는 표현을 쓰지 않는다.
- "해야 한다"보다 "보인다", "모였다", "들어갔다", "펼쳐졌다"를 우선한다.
- 숫자는 문장을 구체화할 때만 사용한다.
- 한 문장은 모바일에서 2줄 이하가 되도록 짧게 쓴다.
- 카피는 실제 데이터에서 확인 가능한 사실만 말한다.

## Out of Scope

- LLM 기반 여행 브리핑 또는 추천
- 사용자 행동 학습 또는 개인화
- 단계 완료 상태 저장
- 온보딩 위저드
- 자동 일정 생성
- 장소 카탈로그 또는 시드 POI
- 새 DB 테이블, 컬럼, 마이그레이션
- 기존 `specs/` 문서 업데이트

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 빈 trip, 후보 중심 trip, 확정 중심 trip, 일정 포함 trip에서 각각 데이터에 맞는 문장이 표시된다.
- **SC-002**: 목록, 지도, 일정 화면 모두에서 동일한 trip 제목 컨텍스트와 함께 Trip Pulse 문장이 보인다.
- **SC-003**: 문장 생성 로직은 최소 10개 케이스의 단위 테스트 또는 동등한 검증 케이스로 확인된다.
- **SC-004**: `npm run lint`와 `npm run build`가 통과한다.
- **SC-005**: 모바일 폭 375px에서 문장과 액션이 겹치거나 잘리지 않는다.
