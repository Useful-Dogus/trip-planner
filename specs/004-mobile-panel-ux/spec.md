# Feature Specification: 모바일 패널 UX 개선

**Feature Branch**: `004-mobile-panel-ux`
**Created**: 2026-03-27
**Status**: Draft
**Input**: User description: "디테일 뷰 및 편집을 위한 패널의 모바일 UX 오류 #52를 수정하고 싶어. 이슈는 당장 제보된 간단한 것들만 기록했어. 실제로 소스코드와 다양한 시나리오를 확인하고 디자인을 수정하고 싶어. 단순히 에러를 최소 수정으로 회피하는 것이 아니라, 이 이슈를 기회로 삼아 더 좋은 UX를 제공할 수 있도록 모바일 UX를 개선하고 싶어."

## Context

현재 `ItemPanel` 컴포넌트는 모바일에서 바텀 시트(Bottom Sheet)로 표시되며, 아이템 상세 보기와 편집 기능을 함께 제공한다. 코드 분석과 다양한 사용 시나리오 검토 결과, 다음과 같은 UX 문제들이 확인되었다:

1. **편집 모드 중 의도치 않은 닫힘**: 스와이프 다운 제스처가 편집 중에도 활성화되어 있어, 폼에서 스크롤하다가 실수로 패널이 닫히고 입력 내용이 사라질 수 있다.
2. **액션 버튼 접근성**: 저장/취소/삭제 버튼이 폼 콘텐츠 맨 아래에 위치해 스크롤 없이 접근하기 어렵다. 모바일에서 가상 키보드가 올라오면 더욱 가려진다.
3. **비저장 변경사항 경고 없음**: 편집 중에 X 버튼이나 백드롭을 탭해서 패널을 닫아도, 입력한 내용이 경고 없이 즉시 사라진다.
4. **링크 편집 레이아웃**: 링크 입력 폼의 2단 그리드(라벨+URL)가 좁은 모바일 화면에서 너무 작아 입력하기 불편하다.
5. **편집 모드 진입 시 포커스 관리 부재**: 편집 버튼 클릭 후 첫 번째 입력 필드에 자동으로 포커스가 이동하지 않아 추가 탭이 필요하다.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 편집 중 실수로 닫기 방지 (Priority: P1)

모바일에서 아이템을 편집하다가 폼에서 스크롤하는 도중 또는 X 버튼을 잘못 탭했을 때, 입력한 내용이 경고 없이 사라지지 않도록 보호한다.

**Why this priority**: 데이터 손실은 가장 치명적인 UX 문제다. 긴 메모나 여러 링크를 입력하다 실수로 닫으면 모든 변경사항이 사라져 사용자 신뢰를 저해한다.

**Independent Test**: 편집 모드에서 임의의 필드를 변경한 뒤, 스와이프 다운 또는 닫기 버튼을 시도했을 때 확인 메시지가 뜨는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 편집 모드에서 아무 필드도 변경하지 않은 상태, **When** 닫기 버튼 또는 백드롭을 탭하면, **Then** 즉시 패널이 닫힌다 (변경사항 없으므로 경고 불필요).
2. **Given** 사용자가 편집 모드에서 하나 이상의 필드를 변경한 상태, **When** 닫기 버튼 또는 백드롭을 탭하면, **Then** "변경사항이 있습니다. 저장하지 않고 나가시겠습니까?" 확인 메시지가 표시된다.
3. **Given** 확인 메시지가 표시된 상태, **When** 사용자가 "나가기"를 선택하면, **Then** 변경사항이 폐기되고 패널이 닫힌다.
4. **Given** 확인 메시지가 표시된 상태, **When** 사용자가 "취소"를 선택하면, **Then** 패널이 열린 채로 유지되고 편집을 계속할 수 있다.
5. **Given** 사용자가 편집 모드에서 폼을 스크롤하는 도중, **When** 위에서 아래로 스와이프하면, **Then** 편집 중에는 스와이프 닫기 제스처가 비활성화된다.

### User Story 2 - 액션 버튼 항상 접근 가능 (Priority: P2)

모바일 편집 화면에서 저장/취소/삭제 버튼이 스크롤 없이, 가상 키보드가 올라와 있을 때도 항상 화면 내에서 접근 가능하다.

**Why this priority**: 핵심 액션(저장, 취소)에 빠르게 접근할 수 없으면 편집 플로우가 깨진다. 가상 키보드로 인해 버튼이 가려지는 문제는 모바일 폼의 전형적인 UX 실패 패턴이다.

**Independent Test**: 편집 모드에서 텍스트 필드를 탭해 가상 키보드를 올린 상태에서, 키보드 위에 저장/취소 버튼이 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 편집 모드에서 폼의 어떤 위치에 있든, **When** 화면을 보면, **Then** 저장/취소 버튼이 항상 패널 하단에 고정되어 스크롤 없이 보인다.
2. **Given** 사용자가 텍스트 입력 필드를 탭해 가상 키보드가 올라온 상태, **When** 키보드가 표시되어 있는 동안, **Then** 저장/취소 버튼이 키보드 바로 위에 보인다.
3. **Given** 저장 버튼이 항상 고정된 상태, **When** 폼 콘텐츠를 스크롤하면, **Then** 폼 내용만 스크롤되고 버튼 영역은 고정된다.

---

### User Story 3 - 링크 편집 레이아웃 개선 (Priority: P3)

모바일 화면에서 링크의 라벨과 URL 입력 필드가 충분한 크기로 표시되어 입력하기 편하다.

**Why this priority**: 링크 추가는 자주 사용하는 기능이고, 현재 2단 그리드 레이아웃은 좁은 화면에서 두 필드 모두 너무 좁아 URL 입력이 불편하다.

**Independent Test**: 링크 추가 후 편집 모드에서 링크 섹션을 보았을 때, 두 필드 각각이 적절한 너비로 표시되고 탭 입력이 편한지 확인한다.

**Acceptance Scenarios**:

1. **Given** 편집 모드에서 링크 섹션이 표시된 상태, **When** 모바일 화면(768px 미만)에서 보면, **Then** 라벨 필드와 URL 필드가 세로로 쌓여(1단 레이아웃) 각각 전체 너비로 표시된다.
2. **Given** 링크 입력이 세로 레이아웃으로 표시된 상태, **When** 링크가 여러 개일 때, **Then** 각 링크 항목이 시각적으로 구분된다.

---

### User Story 4 - 편집 모드 진입 시 포커스 관리 (Priority: P4)

편집 버튼을 탭하면 첫 번째 입력 필드(이름)에 자동으로 포커스가 이동하여, 추가 탭 없이 바로 입력을 시작할 수 있다.

**Why this priority**: 편집 시작 시 첫 필드 자동 포커스는 UX 마찰을 줄이는 작은 개선이다. P4로 낮게 설정한 이유는 모바일 브라우저 제약으로 인해 자동 키보드 표시가 오히려 레이아웃을 방해할 수 있어 트레이드오프가 있기 때문이다.

**Independent Test**: 상세 뷰에서 편집 버튼을 탭한 직후, 이름 필드에 커서가 있는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 상세 뷰에서 편집 버튼을 탭하면, **Then** 편집 폼이 열리면서 이름 입력 필드에 포커스가 위치한다.

---

### Edge Cases

- 편집 중 네트워크 오류로 저장 실패 시, 패널은 열린 채 유지되고 에러 메시지가 표시된다.
- 폼을 변경하다가 저장 후 다시 열었을 때, 패널은 다시 상세 보기(view) 모드로 시작한다.
- 편집 중 기기가 가로 방향으로 회전하는 경우, 패널 레이아웃이 깨지지 않고 정상 표시된다.
- 링크가 0개일 때 링크 섹션은 비어있는 상태로 "링크 추가" 버튼만 보인다.
- 삭제 확인 다이얼로그는 편집 모드의 미저장 변경사항 경고와 별개로 작동한다.
- 변경사항 감지 시 링크 배열은 URL이 비어있지 않은 항목만 비교 대상으로 한다. 빈 링크 행(URL 미입력)만 추가된 경우에는 dirty 상태로 간주하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 편집 모드에서 원래 값과 다른 변경사항이 존재할 때, 닫기(X 버튼, 백드롭 탭, 스와이프) 시도 시 패널 내부 인라인 확인 UI를 표시해야 한다. 인라인 확인 UI는 패널 하단에 "나가기"와 "계속 편집" 버튼이 나타나는 형태로, 네이티브 `confirm()` 팝업을 사용하지 않는다.
- **FR-002**: 편집 모드에서 변경사항이 없을 때(원래 값과 동일), 닫기 시도 시 즉시 닫혀야 한다.
- **FR-003**: 편집 모드에서 스와이프 다운 닫기 제스처는 비활성화되어야 한다.
- **FR-004**: 편집 폼의 저장/취소 버튼 영역은 패널 하단에 고정(sticky)되어, 폼 콘텐츠 스크롤과 관계없이 항상 표시되어야 한다.
- **FR-005**: 삭제 버튼은 편집 모드일 때 패널 헤더 우측(X 버튼 왼쪽)에 배치되며, sticky footer의 저장/취소 버튼과 분리된다. 이를 통해 파괴적 액션과 편집 완료 액션이 명확히 구분된다.
- **FR-006**: 모바일에서 링크 편집 레이아웃은 라벨과 URL이 세로로 쌓이는 1단 구조를 사용해야 한다.
- **FR-007**: 편집 모드 진입 시 이름 입력 필드에 자동 포커스가 설정되어야 한다.
- **FR-008**: 편집 폼 콘텐츠 영역은 독립적으로 스크롤 가능하며, 고정 버튼 영역과 패널 헤더는 스크롤 범위 밖에 위치해야 한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 편집 중 X 버튼이나 백드롭 탭으로 인한 의도치 않은 데이터 손실이 0건 — 변경사항이 있을 때는 반드시 확인 단계를 거친다.
- **SC-002**: 편집 폼의 모든 모드(가상 키보드 숨김/표시 상태 모두)에서 저장 및 취소 버튼을 스크롤 없이 탭할 수 있다.
- **SC-003**: 링크 편집 시 라벨 및 URL 필드가 각각 전체 너비로 표시되어, 작은 화면에서도 오입력 없이 URL을 입력할 수 있다.
- **SC-004**: 편집 버튼 탭 후 추가 탭 없이 바로 이름 필드에 텍스트를 입력할 수 있다.
- **SC-005**: 모바일 기준(375px 뷰포트)에서 패널 전체 동작이 정상적으로 수행된다: 열기, 상세 보기, 편집, 저장, 취소, 삭제.

## Assumptions

- 모바일 기준 뷰포트는 Tailwind의 `md` 브레이크포인트(768px) 미만으로 정의한다.
- 변경사항 감지는 각 필드 값이 원래 아이템 값과 다른지 얕은 비교(shallow compare)로 판단한다. 링크 배열은 URL이 비어있는 항목을 제외한 뒤 비교한다.
- 데스크탑의 경우 현재 사이드 패널 동작은 이 기능의 범위 밖이다 (모바일 바텀 시트만 개선).
- 편집 모드의 자동 포커스는 모바일 사파리의 제약(사용자 제스처 없이는 키보드가 자동으로 열리지 않을 수 있음)을 고려하되, 가능한 범위 내에서 구현한다.
