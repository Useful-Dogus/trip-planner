## Overview

본인과 아내 두 명이 사용하는 뉴욕 10일 여행 플래너 웹 앱. 리서치(후보 장소 조사)와 일정(확정 일정 확인)을 하나의 항목 테이블로 통합 관리한다. 맥북과 아이폰 브라우저에서 모두 동작해야 한다.

**인프라 구조**: 로컬 맥북에서 서버를 실행하고, ngrok으로 외부(아이폰) 접속을 허용한다. 데이터는 GitHub 프라이빗 레포 안의 JSON 파일로 관리한다. Supabase·Vercel 등 외부 서비스는 사용하지 않는다.

**데이터 관리 방식**:
- 앱에서 항목을 추가/수정/삭제하면 서버가 JSON 파일을 직접 수정한다.
- 개발자는 JSON 파일을 텍스트 에디터로 직접 편집하여 데이터를 일괄 변경할 수 있다.
- JSON 파일은 코드와 함께 GitHub 프라이빗 레포에 커밋하여 이력을 관리한다.

## Implementation Responsibilities

### 에이전트(AI)가 처리할 수 있는 작업

- 전체 페이지 라우팅 구조 및 레이아웃 구현
- 로그인 폼 및 인증 처리 로직 구현
- JSON 파일 읽기/쓰기 API 엔드포인트 구현
- 항목 목록 UI (카테고리/상태/우선순위 필터 포함)
- 항목 추가/수정/삭제 폼 구현
- 주소 입력 시 좌표 자동 변환 기능 구현 (무료 지오코딩 API 사용)
- 리서치 뷰 지도(핀, 카테고리별 색상, 클릭 팝업) 구현
- 일정 뷰 목록 (날짜별 그룹, 시간순 정렬) 구현
- 일정 뷰 지도 (날짜 선택, 번호 핀, 동선 연결) 구현
- 초기 데이터 JSON 파일 구조 생성 및 테스트용 샘플 데이터 작성 (카테고리·상태·우선순위 조합을 고루 포함한 뉴욕 가짜 항목 10~15개)
- 상태·우선순위 배지 컴포넌트 구현
- 링크 복수 입력(label + url) UI 구현
- 반응형 레이아웃 (모바일/데스크탑)
- `.env.example` 작성 (ngrok 도메인, JWT 시크릿 등)
- ngrok 실행 방법 및 로컬 서버 시작 스크립트 작성 (`README.md`)

## User Scenarios & Testing

### User Story 1 - 로그인 및 세션 유지 (Priority: P1)

사용자가 ID/PW를 입력해 로그인하면 세션이 유지되어 여행 플래너에 접근할 수 있다. 인증되지 않은 접근은 로그인 페이지로 리다이렉트된다. 로그아웃 시 세션이 종료된다.

**Why this priority**: 인증 없이는 어떤 기능도 사용할 수 없는 진입점이다.

**Independent Test**: 올바른 ID/PW 입력 후 `/research` 접근 가능, 잘못된 자격증명 시 오류 메시지 확인, 로그아웃 후 `/research` 접근 시 `/login` 리다이렉트 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 `/login` 페이지에 있을 때, **When** 올바른 ID/PW를 입력하고 로그인 버튼을 누르면, **Then** `/research` 페이지로 이동하고 이후 모든 보호된 경로에 접근 가능하다.
2. **Given** 사용자가 `/login` 페이지에 있을 때, **When** 잘못된 ID/PW를 입력하면, **Then** 오류 메시지가 표시되고 페이지에 머문다.
3. **Given** 로그인된 사용자가 있을 때, **When** 로그아웃 버튼을 누르면, **Then** 세션이 종료되고 `/login` 페이지로 이동한다.
4. **Given** 로그인하지 않은 사용자가 있을 때, **When** `/research`, `/schedule`, `/items/new` 등 보호된 경로에 직접 접근하면, **Then** 자동으로 `/login` 으로 리다이렉트된다.

### User Story 2 - 여행 항목 추가 (Priority: P1)

사용자가 방문하고 싶은 장소(식당, 관광지, 숙소 등)를 새 항목으로 등록할 수 있다. 이름, 카테고리, 상태, 우선순위, 주소, 링크, 예산, 메모, 날짜/시간을 입력한다. 주소를 입력하면 지도 좌표가 자동으로 채워진다.

**Why this priority**: 항목 추가 없이는 리서치도 일정도 존재하지 않는다. 가장 핵심 동작이다.

**Independent Test**: `/items/new` 에서 항목을 하나 추가하고, 리서치 목록에서 해당 항목을 확인.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자가 `/items/new` 에 있을 때, **When** 이름·카테고리·상태를 입력하고 저장하면, **Then** 항목이 JSON 파일에 저장되고 `/research` 목록에 표시된다.
2. **Given** 항목 추가 폼에서, **When** 주소를 입력하면, **Then** 해당 주소의 위도/경도가 자동으로 채워진다.
3. **Given** 항목 추가 폼에서, **When** 링크를 하나 이상 추가(label + URL)하면, **Then** 저장 후 항목 상세에서 링크 목록이 표시된다.
4. **Given** 항목 추가 폼에서, **When** 필수 필드(이름, 카테고리, 상태)를 비우고 저장하면, **Then** 저장되지 않고 유효성 오류 메시지가 표시된다.

---

### User Story 3 - 항목 수정 및 삭제 (Priority: P1)

사용자가 기존 항목의 내용을 수정하거나 삭제할 수 있다. 특히 상태를 "검토중" → "확정"으로 바꾸는 흐름이 핵심이다.

**Why this priority**: 여행 준비는 지속적으로 상태를 업데이트하는 과정이다. 수정 없이는 실용적이지 않다.

**Independent Test**: 기존 항목의 상태를 "검토중"에서 "확정"으로 변경하고, 일정 뷰에 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 항목 상세 페이지 `/items/[id]` 에 있을 때, **When** 내용을 수정하고 저장하면, **Then** JSON 파일에 변경 내용이 저장되고 목록에 반영된다.
2. **Given** 항목의 상태가 "검토중"일 때, **When** 상태를 "확정"으로 바꾸고 날짜/시간을 입력해 저장하면, **Then** 해당 항목이 일정 뷰 목록에 표시된다.
3. **Given** 항목 상세 페이지에 있을 때, **When** 삭제 버튼을 누르고 확인하면, **Then** JSON 파일에서 항목이 제거되고 목록에서 사라진다.
4. **Given** 항목이 "탈락" 상태일 때, **When** 리서치 지도를 열면, **Then** 해당 항목의 핀이 지도에 표시되지 않는다.

---

### User Story 4 - 리서치 뷰: 목록 조회 및 필터링 (Priority: P2)

사용자가 `/research` 에서 전체 항목을 목록으로 보고, 카테고리·상태·우선순위로 필터링할 수 있다. 항목 클릭 시 상세/수정 화면으로 이동한다.

**Why this priority**: 항목이 많아질수록 필터링이 필수다. P1 기능들이 모두 동작하면 바로 필요해진다.

**Independent Test**: 여러 항목이 등록된 상태에서, 카테고리 "식당" 필터 적용 후 식당 항목만 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 여러 항목이 등록된 상태에서, **When** `/research` 목록 탭을 보면, **Then** 전체 항목(탈락 포함)이 목록으로 표시된다.
2. **Given** 리서치 목록에서, **When** 카테고리 필터로 "식당"을 선택하면, **Then** 식당 항목만 표시된다.
3. **Given** 리서치 목록에서, **When** 상태 필터로 "확정"을 선택하면, **Then** 확정 항목만 표시된다.
4. **Given** 리서치 목록에서, **When** 우선순위 필터로 "반드시"를 선택하면, **Then** "반드시" 항목만 표시된다.
5. **Given** 리서치 목록에서, **When** 항목 하나를 클릭하면, **Then** 해당 항목의 상세/수정 페이지로 이동한다.

---

### User Story 5 - 리서치 뷰: 지도 조회 (Priority: P2)

사용자가 리서치 지도 탭에서 탈락 제외 전체 항목을 지도 핀으로 볼 수 있다. 핀은 카테고리별로 다른 색상이며, 클릭 시 요약 팝업이 표시된다.

**Why this priority**: 장소 간 거리와 위치를 파악하는 것이 뉴욕 여행 계획의 핵심이다.

**Independent Test**: 좌표가 등록된 항목 여러 개를 추가하고, 지도 탭 전환 시 핀이 올바른 위치에 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 좌표가 있는 항목들이 등록된 상태에서, **When** 리서치 지도 탭을 열면, **Then** 탈락 제외 모든 항목이 지도 위에 핀으로 표시된다.
2. **Given** 리서치 지도에서, **When** 식당과 관광지 항목이 모두 있을 때, **Then** 카테고리별로 핀 색상이 다르게 표시된다.
3. **Given** 리서치 지도에서, **When** 핀을 클릭하면, **Then** 항목 이름·카테고리·상태·우선순위가 담긴 요약 팝업이 표시된다.
4. **Given** 리서치 지도에서, **When** "탈락" 상태 항목이 있을 때, **Then** 해당 항목의 핀은 지도에 표시되지 않는다.

---

### User Story 6 - 일정 뷰: 날짜별 목록 조회 (Priority: P3)

사용자가 `/schedule` 에서 상태가 "확정"이고 날짜가 지정된 항목만 날짜별로 그룹화되어 시간순으로 표시되는 일정 목록을 볼 수 있다.

**Why this priority**: 확정 일정이 쌓인 뒤에 필요한 기능이다. MVP에 포함되나 P2 기능 이후에 유용해진다.

**Independent Test**: 서로 다른 날짜에 확정 항목을 여러 개 만들고, 일정 목록 탭에서 날짜별 그룹과 시간순 정렬을 확인.

**Acceptance Scenarios**:

1. **Given** 확정+날짜 항목이 여러 날짜에 걸쳐 있을 때, **When** 일정 목록 탭을 열면, **Then** 날짜별로 그룹화되어 날짜 오름차순으로 표시된다.
2. **Given** 같은 날짜에 시간이 다른 여러 확정 항목이 있을 때, **When** 일정 목록을 보면, **Then** 해당 날짜 그룹 내에서 시작 시간 오름차순으로 정렬된다.
3. **Given** 확정이지만 날짜 미입력 항목이 있을 때, **When** 일정 목록을 보면, **Then** 해당 항목은 일정 목록에 표시되지 않는다.
4. **Given** "확정" 외 다른 상태 항목이 있을 때, **When** 일정 목록을 보면, **Then** 그 항목은 일정 목록에 표시되지 않는다.

---

### User Story 7 - 일정 뷰: 날짜별 지도 동선 확인 (Priority: P3)

사용자가 일정 지도 탭에서 특정 날짜를 선택하면 해당 날짜의 확정 장소들이 번호 핀으로 표시되고 방문 순서대로 선으로 연결되어 동선을 시각적으로 파악할 수 있다.

**Why this priority**: 하루 동선을 미리 파악하는 것은 유용하지만, 목록 뷰가 먼저 안정화되어야 한다.

**Independent Test**: 특정 날짜에 확정 항목 3개를 시간순으로 만들고, 일정 지도에서 해당 날짜 선택 시 번호 핀 3개와 연결선이 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 특정 날짜에 확정 항목이 3개 이상 있을 때, **When** 일정 지도에서 해당 날짜를 선택하면, **Then** 해당 항목들이 시간순 번호 핀(1, 2, 3...)으로 지도에 표시된다.
2. **Given** 일정 지도에서 날짜를 선택했을 때, **When** 해당 날짜에 여러 핀이 표시되면, **Then** 방문 순서대로 핀들이 선(동선)으로 연결된다.
3. **Given** 일정 지도에서, **When** 다른 날짜를 선택하면, **Then** 지도가 해당 날짜의 항목들로 업데이트된다.

---

### Edge Cases

- 주소가 인식되지 않는 경우 좌표가 빈 값으로 저장되고, 해당 항목은 지도에 표시되지 않는다.
- 날짜는 지정했지만 시작 시간이 없는 확정 항목은 일정 목록에서 해당 날짜 그룹의 가장 마지막에 표시된다.
- 링크 URL 형식이 잘못된 경우 저장 전 유효성 오류를 표시한다.
- 동일 날짜의 일정 지도에서 좌표 없는 항목은 지도 핀에서 제외되지만 목록에는 표시된다.
- 브라우저 세션 만료 시 다음 요청에서 로그인 페이지로 리다이렉트한다.
- 아이폰 브라우저에서 지도 스크롤과 페이지 스크롤이 충돌하지 않아야 한다.
- JSON 파일을 개발자가 직접 편집하는 중에 앱에서 동시에 저장 요청이 오면, 서버는 가장 마지막 쓰기를 기준으로 저장한다 (동시성 보장 없음 — 2인 소규모 사용 기준으로 허용).
- ngrok이 꺼진 상태에서 아이폰으로 접속하면 연결 실패 화면이 표시된다 (앱 차원의 처리 없음).

## Requirements

### Functional Requirements

**인증**

- **FR-001**: 시스템은 환경 변수에 설정된 ID/PW 계정으로만 로그인을 허용해야 한다. 회원가입 기능은 제공하지 않는다.
- **FR-002**: 시스템은 로그인 성공 시 세션 토큰을 발급하고, 이후 모든 요청에서 인증 여부를 확인해야 한다.
- **FR-003**: 시스템은 인증되지 않은 사용자의 보호된 경로 접근을 `/login` 으로 리다이렉트해야 한다.
- **FR-004**: 사용자는 언제든 로그아웃하여 세션을 종료할 수 있어야 한다.

**데이터 저장**

- **FR-005**: 모든 여행 항목은 로컬 파일 시스템의 JSON 파일에 저장된다.
- **FR-006**: 항목 추가/수정/삭제 시 서버가 JSON 파일을 즉시 업데이트해야 한다.
- **FR-007**: JSON 파일은 사람이 읽고 편집할 수 있는 형식(들여쓰기 포함)으로 저장되어야 한다.

**항목 관리**

- **FR-008**: 사용자는 여행 항목을 추가, 수정, 삭제할 수 있어야 한다.
- **FR-009**: 각 항목은 이름(필수), 카테고리(필수), 상태(필수), 우선순위, 주소, 좌표, 링크 목록, 예산, 메모, 방문 날짜, 시작 시간을 가진다.
- **FR-010**: 카테고리는 교통·숙소·식당·관광·쇼핑·기타 중 하나여야 한다.
- **FR-011**: 상태는 검토중·보류·대기중·확정·탈락 중 하나여야 한다.
- **FR-012**: 우선순위는 반드시·들를만해·시간 남으면 중 하나여야 한다.
- **FR-013**: 주소를 입력하면 시스템이 해당 주소의 위도/경도 좌표를 자동으로 조회해 저장해야 한다.
- **FR-014**: 링크는 라벨과 URL의 쌍으로 구성되며, 하나의 항목에 여러 링크를 추가할 수 있어야 한다.
- **FR-015**: 메모 필드는 자유 텍스트를 지원해야 한다.

**리서치 뷰**

- **FR-016**: 리서치 뷰는 목록 탭과 지도 탭으로 전환할 수 있어야 한다.
- **FR-017**: 목록 탭은 카테고리, 상태, 우선순위 필터를 제공해야 한다. 필터는 조합 적용이 가능해야 한다.
- **FR-018**: 지도 탭은 탈락 상태를 제외한 모든 항목을 좌표 기반 핀으로 표시해야 한다.
- **FR-019**: 지도 핀은 카테고리별로 다른 색상으로 구분되어야 한다.
- **FR-020**: 지도 핀 클릭 시 항목 이름, 카테고리, 상태, 우선순위를 담은 요약 팝업이 표시되어야 한다.

**일정 뷰**

- **FR-021**: 일정 뷰에는 상태가 "확정"이고 날짜가 입력된 항목만 표시되어야 한다.
- **FR-022**: 일정 목록 탭은 항목을 날짜별로 그룹화하고, 같은 날짜 내에서는 시작 시간 오름차순으로 정렬해야 한다.
- **FR-023**: 일정 지도 탭은 날짜 선택 UI를 제공하고, 선택된 날짜의 항목을 번호 핀으로 표시해야 한다.
- **FR-024**: 일정 지도에서 번호 핀은 해당 날짜의 방문 시간 순서를 나타내며, 핀들은 순서대로 선으로 연결되어야 한다.

**공통 UI**

- **FR-025**: 앱은 맥북(데스크탑) 브라우저와 아이폰(모바일) 브라우저 모두에서 정상 동작하는 반응형 레이아웃이어야 한다.

### Key Entities

- **TripItem (여행 항목)**: 리서치 대상 또는 확정 일정이 된 장소/활동을 나타낸다. 이름, 카테고리, 상태, 우선순위, 위치(주소+좌표), 링크 목록, 예산, 메모, 날짜/시간 정보를 포함한다. 모든 데이터의 중심 엔티티. JSON 파일에 배열로 저장된다.
- **Link (링크)**: 항목에 속하는 참고 URL. 라벨과 URL로 구성되며, 하나의 항목은 복수의 링크를 가질 수 있다.

## Success Criteria

### Measurable Outcomes

- **SC-001**: 새 여행 항목을 추가하는 데 2분 이내에 완료할 수 있다 (항목 이름, 카테고리, 상태, 주소 입력 기준).
- **SC-002**: 리서치 목록에서 필터 적용 후 원하는 항목을 찾는 데 30초 이내에 가능하다.
- **SC-003**: 일정 지도에서 특정 날짜의 동선을 확인하는 데 10초 이내에 가능하다.
- **SC-004**: 상태 변경(검토중 → 확정)을 포함한 항목 수정을 1분 이내에 완료할 수 있다.
- **SC-005**: 아이폰 Safari 브라우저(ngrok URL 접속)에서 모든 주요 기능(로그인, 항목 추가/수정, 목록/지도 조회)이 오류 없이 동작한다.
- **SC-006**: 뉴욕 10일 여행 기준 최소 50개 항목 등록 시에도 목록과 지도가 지연 없이 표시된다.
- **SC-007**: JSON 파일을 텍스트 에디터로 직접 편집한 후 앱을 새로고침하면 변경 내용이 즉시 반영된다.

## Assumptions

- 동시 접속자는 최대 2명(본인·아내)이므로 동시성 충돌은 허용 범위로 본다.
- 맥북이 켜져 있고 ngrok이 실행 중인 경우에만 아이폰 접속이 가능하다. 이는 의도된 제약이다.
- ngrok 무료 플랜을 사용하며, URL이 재시작마다 바뀌는 것을 감수한다.
- 이미지 첨부는 MVP 범위에서 제외한다. 메모는 텍스트만 지원한다.
- 주소-좌표 변환은 뉴욕 지역 주소 위주로 동작하면 충분하다. 무료 지오코딩 API(Nominatim 등) 사용.
- 여행 기간은 10일로 고정이므로 날짜 선택 UI는 해당 기간에 최적화한다.
- 데이터 삭제는 복구 없이 즉시 JSON에서 제거한다. Git 이력로 복구 가능하면 충분하다.
- 다국어 지원 불필요. 한국어 UI만 제공한다.
- 지오코딩 실패 시 사용자가 수동으로 좌표를 입력하는 fallback은 MVP 범위 밖이다.
