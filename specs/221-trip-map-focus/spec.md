# Feature Specification: Trip 단위 지도 자동 도시 포커싱

**Feature Branch**: `221-trip-map-focus`
**Created**: 2026-06-01
**Updated**: 2026-06-01 (리뷰 반영 — 침묵 동작 가시화 / 명시 좌표 보호 / 매칭 false positive 최소화)
**Status**: Draft
**Input**: GitHub Issue [#221](https://github.com/Useful-Dogus/trip-planner/issues/221)

## 배경

현재 trip을 열어도 해당 도시로 지도가 자동 포커싱되지 않는다. 등록된 items가 있으면 그 핀들에 맞춰 `fitBounds`로 좁혀지고, items가 없으면 `region` 텍스트를 매 호출 geocode한 뒤 실패 시 일본 중앙 좌표(`[36.2, 138.25]`, zoom 5)로 떨어진다. 단일 trip(2026-07 NYC) 시대의 잔재.

사용자는 trip 이름·기간·지역을 보고 "이 여행이 어디에 대한 것인지" 멘탈모델을 가진다. 지도가 trip의 도시 정체성을 반영하지 못하면 ("뉴욕 여행인데 왜 도쿄가 보이지?"), 첫인상부터 신뢰가 무너진다.

추가로 다음 침묵 동작이 사용자 신뢰를 흔든다는 점을 1차 리뷰에서 확인했다.

- 폼에서 region 입력 시 사전 매칭 또는 지오코딩 결과가 화면에 표시되지 않아 "어느 도시로 인식됐는지" 확인 불가.
- 지오코딩 실패가 폼 저장 성공으로 위장되어 사용자가 fallback 뷰를 나중에 보고서야 인지.
- 사용자가 "여기를 중심으로"로 명시 저장한 좌표가 region 텍스트 수정만으로 침묵 삭제되어 의도와 다른 위치로 점프.
- 명시 좌표 상태와 "자동으로 되돌리기" 진입점이 발견 불가.
- preset substring 매칭의 false positive("후쿠오카여행" → "오카" 매칭) 위험.
- trip 좌표가 items 좌표보다 우선될 때 items가 viewport 밖에 있으면 사용자는 "내 핀이 사라졌나?" 의문.

이 갱신은 위 6개 항목을 명시성·편집가능성 관점에서 닫는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 주요 도시 trip의 즉시 도시 포커싱 (Priority: P1)

여행자가 "도쿄 여행", "뉴욕 여행", "서울 여행"처럼 한국인이 자주 가는 도시로 trip을 만들고 처음 지도를 열었을 때, 별도 입력 없이 해당 도시 중심부가 도시 전체를 한 눈에 볼 수 있는 줌 레벨로 표시된다.

**Why this priority**: 모든 trip의 첫 인상. 다른 모든 시나리오(items 추가, basecamp 설정, 일정 짜기)는 지도가 올바른 도시에 위치한다는 전제 위에서 작동한다. 이게 깨지면 사용자는 매번 직접 지도를 드래그·줌해서 도시를 찾아야 한다.

**Independent Test**: 도쿄·서울·뉴욕·런던·파리 등 사전 수록 도시 5개로 빈 trip을 각각 만들고 첫 지도 로드 시 해당 도시 중심부가 표시되는지 확인. items 없이도 동작해야 한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "도쿄 여행" trip을 새로 만들고 region 필드에 "도쿄"를 입력해 저장했을 때, **When** trip의 지도 탭을 열면, **Then** 지도 중심은 도쿄 도심(시부야·신주쿠 일대가 화면에 들어오는 도시 레벨) 좌표에 위치한다.
2. **Given** "뉴욕 여행" trip이 있고 등록된 items가 0개일 때, **When** 지도를 열면, **Then** 맨해튼·브루클린이 화면에 보이는 도시 레벨 포커싱이 유지된다.
3. **Given** "서울 여행"에 items가 1개(예: 인천공항)만 등록되어 있을 때, **When** 지도를 열면, **Then** 인천공항 단일 핀에 줌인되지 않고 서울 도시 레벨이 유지되어 다음 핀 추가 위치를 가늠하기 쉽다.
4. **Given** 사용자가 region을 "Tokyo", "도쿄", "동경" 중 어느 표기로 입력해도, **When** 지도를 열면, **Then** 동일하게 도쿄 도심으로 포커싱된다.

### User Story 2 - 사전 미수록 도시도 region 입력 시점에 자동 포커싱 좌표 확보 (Priority: P1)

여행자가 "시드니", "헬싱키", "타슈켄트"처럼 한국인 빈도가 낮아 도시 사전에 포함되지 않은 도시로 trip을 만들 때, trip 생성·편집 폼에서 region 입력을 마치면 그 시점에 외부 지오코딩 1회로 좌표를 확보해 trip 데이터에 저장한다. 이후 지도 열람 시 외부 호출 없이 즉시 표시된다.

**Why this priority**: 사전 수록 도시 약 105개로 다수 여행을 커버하지만 나머지가 막히면 "이 사이트는 유명 도시만 지원한다"는 인상이 박힌다. 외부 호출 시점을 폼 입력 직후 1회로 한정하면 지도 로드 경로는 순수 로컬 연산만 남고, 외부 API 의존이 사용자 경험을 흔들지 않는다.

**Independent Test**: 사전에 없는 도시(예: 시드니) trip을 만들고 폼에서 region "시드니" 입력 후 저장한다. trip 데이터에 좌표가 채워졌는지 확인하고, 지도 열람 시 외부 호출 없이 시드니 포커싱이 되는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 trip 생성 폼에서 region "시드니"를 입력하고 입력란을 벗어났을 때, **When** preset 사전에 매칭되지 않으면, **Then** 외부 지오코딩 1회를 호출해 받은 좌표·줌을 폼 상태에 저장하고, trip 저장 시 함께 영구 저장한다.
2. **Given** 시드니 trip에 좌표가 저장되었을 때, **When** 지도를 열면, **Then** 외부 호출 없이 즉시 시드니로 포커싱된다.
3. **Given** region에 "asdfasdf"처럼 지오코딩이 실패할 문자열을 입력했을 때, **When** 폼을 저장하면, **Then** 좌표는 비워진 채 저장되고 지도를 열면 fallback 뷰가 표시된다. 사용자는 지도에서 직접 도시를 찾아 "여기를 중심으로"(US4) 액션으로 보정할 수 있다. **단**, 폼 저장 전 입력란 아래 inline 메시지로 실패 사실이 명시된다(US6 참조).
4. **Given** 외부 지오코딩 호출이 일시 장애로 실패했을 때, **When** 폼을 저장하면, **Then** region 텍스트만 저장되고 좌표는 비워진다. 사용자가 다음 편집 시점에 region을 다시 저장하면 재시도된다(자동 재시도 X). 일시 장애 여부는 inline 메시지에서 "일시적 장애" 문구로 영구 실패와 구분된다.

### User Story 3 - 잘못 저장된 좌표의 복구 (Priority: P2)

사용자가 region을 수정하면 다음 지도 열람 시 새 region 기준으로 좌표가 다시 계산되어 trip에 덮어쓰기 저장된다. 별도의 "지도 리셋" 버튼 없이 region 텍스트만으로 도시 정체성이 단일 출처에서 관리된다.

**Why this priority**: 1회차에 잘못된 도시(동명이도시·오타·외부 API 일시 오류)로 저장된 좌표를 사용자가 영구히 안고 가지 않도록 복구 경로가 필요하다. 다만 P1 시나리오가 정상 동작하면 잘못 저장될 빈도가 낮으므로 우선순위는 P2.

**Independent Test**: region을 "서울"로 한 trip의 좌표를 강제로 도쿄로 바꾼 뒤, region을 "Seoul"로 다시 저장. 다음 지도 열람 시 서울로 복귀하는지 확인.

**Acceptance Scenarios**:

1. **Given** 좌표가 이미 저장된 trip의 region을 사용자가 다른 값으로 수정했을 때(명시 좌표 아님), **When** 저장하고 지도를 다시 열면, **Then** 새 region에 맞는 좌표로 trip 데이터가 갱신되고 새 도시로 포커싱된다.
2. **Given** trip을 편집 폼에서 region을 비웠을 때, **When** 저장하고 지도를 열면, **Then** 좌표도 비워지고 fallback 뷰(또는 items가 있다면 items 기반 뷰)로 떨어진다.

### User Story 4 - 지도에서 직접 중심점 지정 (Priority: P1)

여행자가 자동 계산 결과(사전 매칭·지오코딩)에 만족하지 않거나, 사전 미수록 도시·멀티-도시 일정처럼 자동으로는 잡기 어려운 중심점이 필요할 때, 지도 화면에서 원하는 위치·줌으로 이동한 뒤 "여기를 중심으로" 액션 하나로 trip의 좌표·줌을 명시 저장한다.

**Why this priority**: 자동 계산은 80% 정도를 커버하지만 다음 케이스에서 사용자가 막힌다 — (1) 사전 도시 좌표가 사용자가 원하는 동네(예: 도쿄역 vs 시부야)와 다를 때, (2) 외부 지오코딩이 동명이도시를 잘못 잡았을 때, (3) 멀티-도시 일정(오사카+교토)에서 중간 지점을 잡고 싶을 때, (4) 숙소가 외곽이라 도시 중심과 일정 동선이 분리될 때. 자동 폴백 + 명시 오버라이드 둘 다 있어야 신뢰성과 자율성이 모두 확보된다.

**Independent Test**: 자동 포커싱된 trip을 열어 임의 위치로 드래그·줌 한 뒤 "여기를 중심으로" 액션 실행. 같은 trip을 닫았다 다시 열어도 그 좌표로 복귀하는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 trip 지도를 열고 원하는 위치·줌으로 조작했을 때, **When** "여기를 중심으로" 액션을 실행하면, **Then** 현재 viewport의 중심 좌표와 줌이 trip 데이터에 저장된다.
2. **Given** 사용자 명시 저장 좌표가 있는 trip을 다시 열었을 때, **When** 지도가 로드되면, **Then** 저장된 좌표·줌으로 즉시 표시되며 자동 계산이 덮어쓰지 않는다.
3. **Given** 사용자 명시 좌표가 저장된 trip에서 region 텍스트를 다른 값으로 수정하려 할 때, **When** 폼에서 region 변경이 감지되면, **Then** 폼은 inline 경고("중심점을 직접 지정했습니다. region을 바꾸면 어떻게 할까요?")와 두 선택지를 노출한다 — "명시 좌표 유지(기본)" / "새 region으로 재계산". 사용자가 명시 선택 없이 저장하면 명시 좌표는 보존된다. (US7-1·2 참조)
4. **Given** 사용자가 "자동으로 되돌리기" 액션을 실행했을 때, **When** 지도를 다시 열면, **Then** trip 저장 좌표가 비워지고 자동 계산 흐름(사전 → 지오코딩 → 폴백)으로 돌아간다.
5. **Given** 사전 미수록 도시이고 외부 지오코딩이 부정확한 좌표를 저장했을 때, **When** 사용자가 정확한 위치에서 "여기를 중심으로"를 실행하면, **Then** 부정확한 좌표가 사용자 의도 좌표로 덮어써진다(복구 경로).

### User Story 5 - items와 basecamp의 일관된 우선순위 (Priority: P2)

trip이 도시 좌표를 보유한 경우, 사용자가 items를 추가하거나 basecamp를 설정하더라도 지도의 초기 도시 정체성은 흔들리지 않는다. items 좌표 군집에 맹목적으로 줌인되어 "어디에 핀을 찍을지" 컨텍스트가 사라지지 않는다.

**Why this priority**: P1에 의존적이지만 기존 사용자(items 이미 등록된 trip)의 멘탈모델 변화를 일관되게 보장하려면 명시적으로 정의되어야 한다. basecamp(숙소)는 외곽에 잡히거나 미정인 경우가 많아 도시 중심 후보로 부적합 — 핀으로만 표시되고 줌·중심 결정에는 사용되지 않는다(US4 명시 좌표 부재 시 fallback 단계에서만 등장).

**Independent Test**: items 1개와 5개가 있는 trip에서 도시 레벨이 유지되는지 vs 군집에 줌인되는지 시각적으로 확인.

**Acceptance Scenarios**:

1. **Given** trip에 도시 좌표가 저장되어 있고 items 1개(공항)가 있을 때, **When** 지도를 열면, **Then** 도시 좌표가 우선되어 도시 레벨 뷰가 표시된다. 공항 핀이 viewport 밖에 위치하면 화면에 "핀 N개가 화면 밖" 인디케이터가 노출되고, 인디케이터 1탭으로 "모든 핀 보기"(items+trip 좌표를 모두 포함하는 fitBounds)로 전환할 수 있다(US8 참조).
2. **Given** trip에 도시 좌표가 있고 items가 5개 이상으로 자연스러운 군집을 이룰 때, **When** 지도를 열면, **Then** trip 좌표가 우선되되 화면 안에 모든 items가 들어오도록 적정 줌으로 보정된다(도시 레벨을 넘어 줌아웃하지 않는다).
3. **Given** trip에 도시 좌표와 basecamp가 모두 설정되어 있을 때, **When** 지도를 열면, **Then** trip 도시 좌표가 우선되고 basecamp는 핀으로만 표시된다.

### User Story 6 - 폼 region 입력 결과 inline 피드백 (Priority: P1)

여행자가 폼에서 region을 입력하고 입력란을 벗어나면, 어느 도시로 인식됐는지(또는 인식 실패했는지) 그 자리에서 명시 확인할 수 있다. 폼을 저장하기 전에 결과를 보고 수정할 수 있어 "저장 후 지도 열어보고 의외 발견" 경로가 사라진다.

**Why this priority**: 침묵 동작 제거의 핵심. 사용자가 도시 정체성을 폼에서 직접 확인·편집할 수 있어야 명시성·편집가능성이 확보된다. P1 시나리오들(US1·US2·US4) 모두 이 피드백이 없으면 신뢰성이 깨진다.

**Independent Test**: 폼에서 "도쿄", "asdfasdf", "서울특별시" 입력 후 blur 시점에 각각 다른 inline 메시지가 노출되는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 폼에서 region "도쿄"를 입력하고 입력란을 벗어났을 때, **When** preset 매칭이 성공하면, **Then** 입력란 아래에 매칭된 도시명 칩("📍 도쿄로 인식됨")과 미니 좌표 미리보기(또는 "지도 미리보기" 토글)가 노출된다.
2. **Given** 사용자가 폼에서 매칭된 도시 칩 "📍 도쿄로 인식됨"을 보고 "다른 도시" 액션을 클릭했을 때, **When** 액션을 실행하면, **Then** preset 매칭이 해제되고 외부 지오코딩을 다시 시도하거나, 사용자가 "지도에서 직접 지정" 진입점으로 이동할 수 있다.
3. **Given** 사용자가 폼에서 region "시드니"를 입력하고 입력란을 벗어났을 때, **When** preset 미매칭으로 외부 지오코딩을 호출해 성공하면, **Then** 입력란 아래에 "📍 Sydney, Australia로 인식됨" 칩과 미리보기가 노출된다. preset 매칭과 외부 결과는 칩에서 출처가 구분된다(예: 보조 라벨 "preset" / "검색 결과").
4. **Given** 사용자가 폼에서 region "asdfasdf"를 입력하고 입력란을 벗어났을 때, **When** preset 미매칭 + 외부 지오코딩도 결과 없음일 때, **Then** 입력란 아래에 "「asdfasdf」 위치를 찾지 못했습니다. 지도에서 직접 지정하거나 다른 표기를 시도하세요" 메시지와 "지도에서 직접 지정" 진입점이 노출된다. 폼 저장은 막지 않는다.
5. **Given** 외부 지오코딩 호출이 네트워크/일시 장애로 실패했을 때, **When** 결과가 반환되면, **Then** 메시지는 영구 실패와 구분되어 "위치 검색이 일시적으로 실패했습니다. 잠시 후 다시 시도하거나 지도에서 직접 지정하세요"로 노출된다. 폼 저장은 막지 않는다.

### User Story 7 - 명시 좌표 상태·복귀 진입점 가시화 (Priority: P1)

사용자 명시 좌표가 활성인 trip에서 지도를 열면 "지금 직접 지정한 중심점을 사용 중"이라는 상태와 자동 계산으로 돌아가는 진입점이 항상 화면에서 한 번에 보인다. 폼에서 region을 바꿀 때도 명시 좌표가 침묵 삭제되지 않도록 의도를 묻는다.

**Why this priority**: US4의 "여기를 중심으로" 액션이 무의미해지는 가장 큰 원인이 (a) 명시 상태인지 사용자가 모름 (b) "되돌리기" 진입점이 어디 있는지 모름 (c) region 한 글자 수정으로 명시 좌표가 사라지는 경험이다. 이 세 침묵을 동시에 닫는다.

**Independent Test**: 명시 좌표 저장 후 지도 재진입 시 상태 배너가 보이는지 확인. region 텍스트 수정 시 폼이 의도를 묻는 inline 경고를 띄우는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자 명시 좌표가 활성인 trip을 열었을 때, **When** 지도가 로드되면, **Then** 지도 상단(또는 명확히 발견 가능한 위치)에 상태 배너("📍 직접 지정한 중심점 사용 중 · [자동으로 되돌리기]")가 노출된다. 자동 계산 좌표를 사용 중인 trip에서는 배너가 표시되지 않는다.
2. **Given** 사용자 명시 좌표가 활성인 trip의 폼에서 region 텍스트가 변경된 것이 감지됐을 때, **When** 폼이 저장 직전 또는 region 입력란 변경 시점에 도달하면, **Then** inline 경고("중심점을 직접 지정했습니다. region을 바꾸면 어떻게 할까요?")와 두 선택지("명시 좌표 유지" / "새 region으로 재계산")가 노출된다. 기본 선택은 "명시 좌표 유지". 사용자가 선택을 변경하지 않은 채 저장하면 명시 좌표는 보존되고 region 텍스트만 갱신된다.
3. **Given** 사용자가 위 경고에서 "새 region으로 재계산"을 선택하고 저장했을 때, **When** 다음 지도 열람이 발생하면, **Then** 새 region 기준으로 좌표가 재계산되어 덮어써지고 상태 배너는 사라진다.
4. **Given** 사용자가 지도 상태 배너에서 "자동으로 되돌리기"를 실행했을 때, **When** 액션이 완료되면, **Then** 명시 좌표가 비워지고 즉시 자동 계산 흐름으로 재포커싱된다. 배너는 사라지고 사용자는 결과 위치를 그 자리에서 확인할 수 있다.

### User Story 8 - viewport 밖 items 가시화 (Priority: P2)

trip 좌표 우선 정책 때문에 일부 items가 viewport 밖에 위치할 때 사용자는 "내 핀이 사라졌나" 의문을 갖는다. 지도에 "핀 N개가 화면 밖" 인디케이터를 노출해 즉시 인지하고 "모든 핀 보기"로 전환할 수 있게 한다.

**Why this priority**: US5의 trip 좌표 우선 정책을 사용자가 안심하고 받아들이게 하는 보완 장치. P1은 아니지만 US5 채택 시 거의 같은 PR에서 함께 닫혀야 의미가 있다.

**Independent Test**: 도쿄 trip에 나리타공항 핀 1개 추가한 뒤 지도를 열어 인디케이터·"모든 핀 보기" 토글이 동작하는지 확인.

**Acceptance Scenarios**:

1. **Given** trip 좌표 기준 viewport에 표시되는 items 핀이 전체 items보다 적을 때, **When** 지도가 로드되면, **Then** 지도 가장자리(또는 floating 배지)에 "핀 N개가 화면 밖" 인디케이터가 노출된다. N은 viewport 밖 items 수.
2. **Given** 인디케이터가 노출된 상태에서, **When** 사용자가 인디케이터를 1탭하면, **Then** trip 좌표를 무시하고 "items + trip 좌표를 모두 포함하는 fitBounds"로 즉시 전환되며 인디케이터는 사라진다. 단, 이 전환은 1회성이며 trip 좌표 정책 자체를 영구 변경하지 않는다.
3. **Given** trip 좌표 viewport에 모든 items 핀이 포함될 때, **When** 지도가 로드되면, **Then** 인디케이터는 표시되지 않는다.

---

### Edge Cases

- **단일 trip 시대 잔재**: 기존 데이터에 region 없이 items만 있는 trip이 남아있을 때, 지도는 items 기반 fitBounds로 작동하며 일본 좌표가 노출되지 않는다.
- **region에 국가명만 입력** ("일본", "미국"): 사전에 없으므로 지오코딩 결과를 수용. 국가 중심 좌표는 도시보다 줌아웃되지만 fallback보다는 적합. 도시로 좁히는 가이드는 별도 이슈(region 입력 시 자동완성).
- **region에 모호한 도시명** ("케임브리지" → 영국 vs 미국): 첫 지오코딩 결과를 그대로 저장. 사용자가 잘못된 도시면 inline 칩(US6-3)에서 결과 도시명을 확인하고 region 텍스트 수정 또는 "지도에서 직접 지정"으로 보정.
- **외부 지오코딩 일시 장애**: 폼 저장 시점에 호출이 실패하면 좌표는 비워진 채 trip이 저장된다. 다음 열람은 fallback 뷰로 떨어지고, 사용자는 region 재저장 또는 "여기를 중심으로"(US4)로 보정한다. 지도 로드 경로에서는 외부 호출이 일어나지 않으므로 외부 API 장애가 지도 로드 자체를 막지 않는다. 일시 장애 / 영구 실패는 US6-4·5의 inline 메시지에서 구분 표기된다.
- **모바일·데스크탑 동등성**: 모든 지도 컴포넌트가 동일한 우선순위를 사용. 뷰별로 다른 포커싱이 일어나지 않는다. 상태 배너·인디케이터도 모바일·데스크탑 양쪽에서 동등하게 노출된다.
- **사용자가 지도를 드래그·줌한 직후**: 자동 포커싱은 trip 로드 1회만 적용. 이후 사용자 조작을 덮어쓰지 않는다(기존 동작 유지). 단, 드래그·줌만으로는 명시 좌표가 저장되지 않으며, "여기를 중심으로" 명시 액션을 통해서만 영속화된다.
- **preset substring false positive**: "후쿠오카여행"에 "오카"(오사카 약어) 매칭 같은 부분 문자열 충돌은 FR-005a로 차단한다.
- **명시 좌표 + 빈 region**: 사용자가 명시 좌표를 둔 채 region을 비우면, 명시 좌표는 보존(US7-2의 "유지" 기본값)되며 도시 정체성 라벨만 사라진다. 상태 배너는 계속 노출.

## Requirements *(mandatory)*

### Functional Requirements

#### 데이터 모델
- **FR-001**: 시스템은 각 trip별로 지도 중심점 좌표(위도·경도)와 기본 줌 레벨을 저장할 수 있어야 한다.
- **FR-002**: 좌표·줌 값은 trip 단위로 격리되어 한 사용자의 trip에 저장된 값이 다른 사용자의 trip에 노출되지 않으며, 한 사용자가 자신의 trip에 저장한 잘못된 좌표가 다른 사용자의 trip 좌표에 영향을 주지 않아야 한다.
- **FR-003**: 좌표·줌 값은 비어 있을 수 있으며, 비어 있을 때는 region 또는 fallback으로 대체 가능해야 한다.

#### 도시 사전(preset)
- **FR-004**: 시스템은 한국인 여행 빈도가 높은 도시 약 105개의 사전 좌표를 내장하며, 각 도시는 한국어·영문·약어 별칭으로 매칭 가능해야 한다.
- **FR-005**: 사전 매칭은 region 입력값을 정규화(소문자화, 공백·특수문자 제거)한 뒤 별칭과 매칭하며, "도쿄여행", "Tokyo trip", "TOKYO" 모두 동일한 도시 항목과 매칭되어야 한다.
- **FR-005a**: 사전 매칭은 false positive를 최소화해야 한다 — 예) "후쿠오카여행"이 "오카"(오사카 별칭)와 매칭되거나, "뉴욕시"가 "욕"으로 매칭되는 것을 방지. 정확한 매칭 알고리즘(token 분리, 정규화 후 exact 우선, 별칭별 가중치 등)은 plan 단계에서 결정하되, 한국어 substring 충돌 케이스를 회귀 테스트로 검증해야 한다.
- **FR-006**: 사전 각 도시 항목은 도시 규모에 맞는 기본 줌 레벨을 가져야 한다(거대 도시 11, 일반 도시 12, 소도시·리조트 12-13).

#### 지도 초기 포커싱 우선순위
- **FR-007**: 지도는 trip 로드 시 다음 우선순위로 초기 뷰를 결정한다 — (1) trip 저장 좌표(자동 또는 사용자 명시) → (2) region preset 매칭 → (3) items fitBounds(items ≥ 1) → (4) basecamp 좌표 → (5) 세계 중심 폴백. 지도 로드 경로는 외부 API 호출을 수행하지 않는다(외부 호출은 trip 폼 입력 시점에만 발생, FR-014).
- **FR-008**: items 기반 fitBounds는 단일 도시 레벨(zoom ≤ 14)을 초과해 줌인하지 않아야 한다.
- **FR-009**: 모든 지도 화면(목록·일정·리서치)이 동일한 우선순위를 사용해야 한다.
- **FR-010**: 폴백 뷰는 단일 국가 좌표를 사용하지 않고 지구 전체를 포괄하는 좌표(예: 위도 20, 경도 0, zoom 2)를 사용해야 한다.

#### 외부 지오코딩 호출 한정
- **FR-011**: 외부 지오코딩은 trip 생성·편집 폼에서 region 입력 완료 시점에만 호출되며, 지도 로드·뷰 전환에서는 호출하지 않는다.
- **FR-012**: preset 사전 매칭이 성공하면 외부 지오코딩을 호출하지 않는다(우선순위: preset → 외부 호출).
- **FR-013**: 외부 지오코딩 실패 시 좌표는 비워서 저장하며, 자동 재시도는 하지 않는다. 사용자가 region을 다시 저장하거나 지도에서 "여기를 중심으로"(FR-019)를 사용해 보정한다. 실패 사실은 폼에 inline으로 명시된다(FR-024).

#### 폼 동작
- **FR-014**: trip 생성·편집 폼은 region 필드 입력 완료 시 자동으로 사전 매칭 또는 외부 지오코딩을 수행해 좌표·줌을 폼 상태에 채워야 한다.
- **FR-015**: 사용자에게 좌표를 텍스트(숫자) 직접 입력 칸으로 노출하지 않는다. 좌표 보정은 지도 상호작용으로만 수행한다(FR-019).
- **FR-016**: region 텍스트 수정은 다음 저장 시 좌표 재계산을 트리거해 잘못 저장된 좌표를 사용자가 복구할 수 있어야 한다. 단, 사용자 명시 좌표(FR-019)가 활성인 경우 region 변경은 FR-026의 의도 확인 플로우를 통과해야 하며, 사용자가 명시 선택을 하지 않으면 명시 좌표는 보존된다.

#### 사용자 명시 좌표 지정
- **FR-019**: 사용자는 trip 지도 화면에서 현재 viewport의 중심 좌표·줌을 trip 데이터에 1탭으로 저장할 수 있어야 한다("여기를 중심으로" 액션).
- **FR-020**: 사용자 명시 좌표가 저장된 trip은 region 변경이 없는 한 자동 계산(사전 매칭·외부 지오코딩)이 좌표를 덮어쓰지 않아야 한다.
- **FR-021**: 사용자는 "자동으로 되돌리기" 액션으로 명시 좌표를 비우고 자동 계산 흐름으로 복귀할 수 있어야 한다.

#### 폼 결과 가시화 (inline 피드백)
- **FR-023**: 폼 region 입력란은 입력 완료(blur) 시점에 매칭·지오코딩 결과를 입력란 아래 inline으로 노출해야 한다. 결과 유형은 다음 4가지로 구분된다 — (a) preset 매칭 성공, (b) 외부 지오코딩 성공, (c) 영구 실패(결과 없음), (d) 일시 장애(네트워크/타임아웃).
- **FR-024**: 매칭/지오코딩 성공 시 매칭된 도시명 칩(예: "📍 도쿄로 인식됨")과 미니 좌표 미리보기(또는 미리보기 토글)를 노출해야 한다. preset 매칭과 외부 지오코딩 결과는 칩의 출처 라벨로 구분되어야 한다.
- **FR-024a**: 매칭된 칩에는 "다른 도시"(또는 동등한) 액션이 노출되어 사용자가 매칭 결과를 거부하고 외부 지오코딩 재시도 또는 "지도에서 직접 지정" 진입점으로 이동할 수 있어야 한다.
- **FR-025**: 영구 실패(c) 시 "「{입력값}」 위치를 찾지 못했습니다. 지도에서 직접 지정하거나 다른 표기를 시도하세요" 메시지와 "지도에서 직접 지정" 진입점을 노출해야 한다. 일시 장애(d) 시 "위치 검색이 일시적으로 실패했습니다" 문구로 영구 실패와 구분해야 한다. 두 경우 모두 폼 저장 자체는 차단하지 않는다.

#### 명시 좌표 상태·복귀 가시화
- **FR-027**: 사용자 명시 좌표가 활성인 trip의 지도 화면에는 상태 배너("📍 직접 지정한 중심점 사용 중 · [자동으로 되돌리기]")가 상시 노출되어야 한다. 자동 계산 좌표를 사용 중인 trip에는 배너가 표시되지 않는다.
- **FR-028**: 상태 배너의 "자동으로 되돌리기"는 FR-021과 동일한 동작을 수행하며, 액션 후 즉시 지도가 자동 계산 결과로 재포커싱되어야 한다.
- **FR-026**: 사용자 명시 좌표가 활성인 trip의 폼에서 region 텍스트가 변경된 것이 감지되면, 폼은 inline 경고("중심점을 직접 지정했습니다. region을 바꾸면 어떻게 할까요?")와 두 선택지("명시 좌표 유지" / "새 region으로 재계산")를 노출해야 한다. 기본값은 "명시 좌표 유지"이며, 사용자가 명시 선택 없이 저장하면 명시 좌표는 보존되고 region 텍스트만 갱신된다.

#### viewport 밖 items 가시화
- **FR-029**: 지도 로드 시 trip 좌표 기준 viewport에 표시되지 않는 items가 존재하면, 화면에 "핀 N개가 화면 밖" 인디케이터를 노출해야 한다.
- **FR-030**: 인디케이터 1탭으로 trip 좌표를 무시하고 "items + trip 좌표를 모두 포함하는 fitBounds"로 1회성 전환할 수 있어야 한다. 이 전환은 trip 좌표 정책 자체를 영구 변경하지 않는다.

### Key Entities

- **Trip 좌표 정보**: trip이 가리키는 도시의 중심 위도·경도, 기본 줌 레벨, 그리고 좌표가 사용자 명시(manual)인지 자동 계산(auto)인지 구분하는 출처 플래그. trip과 1:1 관계. 비어 있을 수 있다. 출처 플래그는 자동 계산이 사용자 명시 좌표를 덮어쓰지 않게 하기 위한 가드, 그리고 지도 상태 배너 노출 여부 결정(FR-027)에 사용된다.
- **City Preset**: 도시 한국어·영문·약어 별칭, 중심 좌표(위도·경도), 기본 줌 레벨을 가진 사전 항목. 시스템 내장(공유 사전 아님, 코드 상수로 관리).
- **Region 입력 결과 상태**: 폼 region 입력에 대한 가장 최근 매칭/지오코딩 결과를 폼 세션 단위로 보유하는 일시 상태. 유형(preset/외부/영구실패/일시장애), 매칭된 도시명, 좌표·줌, 출처 라벨을 포함한다. trip 저장 시 좌표·줌·출처가 trip 데이터로 영속화되고, 표시 상태(칩·메시지)는 폼 세션 종료 시 소멸한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사전 수록 도시(한국·일본·중화권·동남아·유럽·북미 등 약 105개) trip 새로 만들 시 첫 지도 열람에서 해당 도시 도시 레벨로 포커싱된다(폴백 또는 잘못된 도시로 떨어지지 않는다).
- **SC-001a**: 사전 매칭 정확도 — 한국어·영문·약어 표기로 입력된 사전 수록 도시명에 대해, 폼 매칭 결과가 사용자가 의도한 도시와 일치하는 비율이 회귀 테스트 케이스(최소 50개 표기 변형) 기준 100%이며, false positive(다른 도시로 잘못 매칭)는 0건이어야 한다.
- **SC-002**: 사전 미수록 도시 trip은 폼 저장 시점에 외부 지오코딩 1회로 좌표를 확보하고, 모든 지도 열람은 외부 호출 0회로 표시된다.
- **SC-003**: 어떤 trip을 열든 일본 중앙 좌표(또는 임의 단일 국가 좌표)가 노출되지 않는다(시각적으로 확인 가능).
- **SC-004**: trip 좌표는 trip 단위로 격리되며 다른 사용자의 trip 표시에 영향을 주지 않는다(공유 사전·테이블 없음).
- **SC-005**: items가 1개만 있는 trip을 열었을 때, 그 단일 핀에 줌인되지 않고 도시 레벨 뷰가 유지된다. items가 viewport 밖에 있을 경우 인디케이터가 노출된다.
- **SC-006**: 모든 지도 화면(목록·일정·리서치)에서 동일한 초기 포커싱이 일어난다.
- **SC-007**: 신규 trip 생성 후 첫 지도 열람까지 사용자가 별도로 좌표·줌을 조작하지 않아도 된다.
- **SC-008**: 사용자가 지도에서 "여기를 중심으로" 액션 1회로 trip 중심점을 변경할 수 있고, 다음 열람부터 자동 계산 결과가 덮어쓰지 않는다.
- **SC-009**: trip 이름에 도시명이 없는 경우(예: "신혼여행", "가족여행")에도 region 입력으로 동일하게 도시 포커싱이 동작한다.
- **SC-010**: 폼 region 입력 시 입력 완료(blur) 후 1초 이내에 매칭/지오코딩 결과가 inline으로 사용자에게 표시된다(성공·실패 모두). 사용자가 폼 저장 전에 결과를 확인할 수 있다.
- **SC-011**: 사용자 명시 좌표가 활성인 trip을 열면 100%의 경우 지도 상단에 상태 배너가 노출되며, 사용자는 1탭으로 "자동으로 되돌리기"에 접근할 수 있다.
- **SC-012**: 사용자 명시 좌표 활성 상태에서 region 텍스트를 변경하고 저장 시도 시 100%의 경우 의도 확인 inline 경고가 노출되며, 명시 선택 없이 저장하면 명시 좌표가 보존된다(침묵 삭제 0건).

## Assumptions

- 사용자가 trip을 만들 때 region 필드를 입력한다는 점은 기존 폼 UX에서 유도되고 있다. 빈 region은 fallback 뷰로 떨어지며, 이는 의도된 동작이다.
- 외부 지오코딩 API의 신뢰성(Nominatim 등)이 99% 이상이며, 일시 장애 시 다음 열람에서 자연 복구된다고 가정한다.
- 도시 사전은 코드 상수로 관리하며, 사용자 요청에 따라 PR로 확장한다. 검색 자동완성 통합 등 동적 사전 필요성은 후속 이슈에서 다룬다.
- 모호한 도시명(예: 영국 케임브리지 vs 미국 케임브리지)의 disambiguation은 외부 지오코딩의 첫 결과를 신뢰하며, inline 칩(FR-024)을 통해 사용자가 결과 도시명을 확인 후 region 텍스트 수정 또는 "지도에서 직접 지정"으로 보정한다.
- 폼 region 결과 inline 피드백의 좌표 미리보기는 별도 지도 인스턴스 또는 정적 미리보기 중 하나로 구현 가능하며, 정확한 형식은 plan 단계에서 결정한다.

## Out of Scope

- 멀티-도시 trip 지원(NYC+필라델피아처럼 trip 1개에 도시 1:N 모델링).
- 공유 `cities` 테이블 및 검색 자동완성 통합.
- 좌표 숫자 직접 입력 UI(위도·경도 텍스트 박스).
- 모호 도시명의 다중 후보 선택 UI(disambiguation).
- trip 이름 자동 파싱으로 region 자동 채움 — 파싱 신뢰도와 다중 출처 충돌 위험이 크며, region 명시 입력이 단일 출처 원칙에 부합하므로 채택하지 않는다.
- Mapbox·3D 지도 이전.
- 사용자의 드래그·줌 결과를 매번 자동 저장(현재 viewport를 매번 영속화)하는 기능 — 명시 액션("여기를 중심으로")으로만 저장한다.
- 단일 trip 시대 데이터의 일괄 backfill SQL(lazy 방식이 안전·정확하므로 채택).
- viewport 밖 items 인디케이터에서 "모든 핀 보기" 전환을 trip 좌표 정책 자체 변경으로 영속화하는 동작(1회성 전환만 허용).
