# Feature Specification: 아이템 메타데이터 UX 재설계

**Feature Branch**: `007-item-metadata-ux`
**Created**: 2026-04-06
**Status**: Draft
**Input**: User description: "카테고리, 상태, 예약상태, 우선순위의 정보 구조와 선택 UX를 재설계한다. 카테고리는 성격 분류로 확장하고, 상태는 의사결정 단계로 단순화하며, 예약상태를 별도 필드로 추가한다. 표시 방식은 칩을 유지하되 값이 없더라도 정보 없음 상태를 보여주고, 선택은 칩 클릭 시 드롭다운으로 변경한다."

## Context

현재 앱은 `category`, `status`, `priority` 3개 필드를 사용하며, 용어와 값 체계가 혼재되어 있다. 특히 `status`는 "의사결정 상태"와 "실행 준비 상태"가 함께 섞여 있어 사용자가 항목의 현재 의미를 빠르게 파악하기 어렵다. 또한 상세 패널과 카드에서 일부 값은 표시되지 않거나, 특정 값만 빠르게 수정 가능해 필드별 UX 일관성이 부족하다.

이번 개편은 단순 라벨 변경이 아니라 다음 구조 변경을 포함한다.

1. `category`를 더 세분화된 "성격 분류" 체계로 확장한다.
2. `status`를 "의사결정 단계"로 재정의한다.
3. `reservation_status`를 신규 필드로 추가해 예약 필요 여부와 완료 여부를 분리한다.
4. `priority`를 유지하되 "선택 강도"로 의미를 명확히 한다. `스킵`은 제거한다.
5. 4개 필드 모두를 칩 형태로 일관되게 표시하고, 값이 없을 때도 비어 보이지 않도록 placeholder 칩을 제공한다.
6. 칩 클릭 시 드롭다운에서 값을 수정하는 상호작용을 공통 패턴으로 제공한다.

## Information Model

### Category (성격 분류)

- 교통
- 숙박
- 명소
- 식당
- 카페
- 쇼핑
- 문화시설
- 공연·스포츠
- 액티비티
- 휴양
- 기타

### Decision Status (의사결정 단계)

- 아이디어
- 검토
- 확정
- 제외

### Reservation Status (실행 조건)

- 확인 필요
- 불필요
- 필요(미예약)
- 예약완료

### Priority (선택 강도)

- 반드시
- 들를만해
- 시간 남으면

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 항목의 성격과 진행 상태를 즉시 파악하기 (Priority: P1)

사용자는 목록 카드와 상세 패널에서 각 항목의 성격 분류, 의사결정 단계, 예약상태, 우선순위를 빠르게 읽고 현재 상황을 혼동 없이 이해할 수 있다.

**Why this priority**: 정보 구조가 명확하지 않으면 이후의 편집 UX 개선도 체감 가치가 낮다. 먼저 "무슨 상태인지 바로 읽히는가"가 해결되어야 한다.

**Independent Test**: 샘플 아이템을 여러 상태 조합으로 준비한 뒤, 목록 카드와 상세 패널에서 4개 필드가 모두 일관된 칩으로 보이고 값이 없을 때도 placeholder가 노출되는지 확인하면 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 항목에 4개 필드 값이 모두 있는 상태, **When** 사용자가 목록 카드 또는 상세 패널을 보면, **Then** 카테고리, 상태, 예약상태, 우선순위가 각각 별도 칩으로 표시된다.
2. **Given** 항목에 `reservation_status` 또는 `priority` 값이 없는 상태, **When** 사용자가 목록 카드 또는 상세 패널을 보면, **Then** 해당 위치에는 숨김 대신 "예약 정보 없음", "우선순위 정보 없음" 같은 placeholder 칩이 표시된다.
3. **Given** 항목의 의사결정 단계가 `확정`이고 예약상태가 `필요(미예약)`인 상태, **When** 사용자가 항목을 보면, **Then** "갈지 말지" 결정과 "예약 실행 여부"를 별개 정보로 읽을 수 있다.
4. **Given** 사용자가 목록 카드를 보는 상태, **When** 메타데이터 칩이 여러 개 표시되면, **Then** 4개 칩 모두 노출되며 한 줄이 넘칠 경우 자연스럽게 줄바꿈된다.

---

### User Story 2 - 칩 클릭으로 메타데이터를 빠르게 수정하기 (Priority: P1)

사용자는 상세 패널과 주요 항목 표현 영역에서 칩을 클릭해 드롭다운을 열고, 전체 편집 폼으로 이동하지 않고도 값을 빠르게 바꿀 수 있다.

**Why this priority**: 이번 기능의 핵심은 "표시만 개선"이 아니라 "읽은 자리에서 즉시 수정"하는 것이다. 클릭 후 드롭다운 수정이 실제 편집 흐름의 중심이다.

**Independent Test**: 상세 패널에서 4개 칩 각각을 클릭해 드롭다운이 열리고, 값을 선택한 직후 UI와 저장 데이터가 갱신되는지 확인하면 독립 검증이 가능하다.

**Acceptance Scenarios**:

1. **Given** 사용자가 상세 패널을 열어본 상태, **When** 카테고리 칩을 클릭하면, **Then** 카테고리 옵션 목록이 드롭다운으로 열린다.
2. **Given** 드롭다운이 열린 상태, **When** 사용자가 새 값을 선택하면, **Then** 선택 즉시 저장이 수행되고 칩 텍스트와 스타일이 갱신된다.
3. **Given** 값이 없는 placeholder 칩이 표시된 상태, **When** 사용자가 해당 칩을 클릭하면, **Then** 실제 값이 있는 칩과 동일한 방식으로 드롭다운이 열려 값을 설정할 수 있다.
4. **Given** 드롭다운이 열린 상태, **When** 사용자가 바깥 영역을 클릭하거나 ESC를 누르면, **Then** 드롭다운이 닫히고 기존 값은 유지된다.

---

### User Story 3 - 신규 분류 체계로 일관되게 생성/편집하기 (Priority: P2)

사용자는 새 항목 생성, 전체 편집 폼, 빠른 편집 UI 어디에서든 동일한 분류 체계와 라벨을 사용해 혼동 없이 항목을 관리할 수 있다.

**Why this priority**: 조회 화면만 바뀌고 생성/편집 폼이 예전 체계를 유지하면 데이터 일관성이 깨진다. 입력 채널 전체가 새 체계를 따라야 한다.

**Independent Test**: 새 항목 생성 폼과 기존 항목 편집 폼에서 4개 필드의 옵션과 라벨이 빠른 편집 UI와 동일한지 확인하면 된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 새 항목 추가 폼을 여는 상태, **When** 분류/상태 관련 입력 영역을 보면, **Then** 카테고리, 상태, 예약상태, 우선순위가 새 용어와 옵션으로 제공된다.
2. **Given** 기존 데이터가 이전 체계로 저장돼 있는 상태, **When** 사용자가 항목을 열면, **Then** 정의된 마이그레이션 규칙에 따라 새 체계 값 또는 placeholder로 안전하게 표시된다.
3. **Given** 사용자가 전체 편집 폼에서 값을 변경해 저장한 상태, **When** 목록과 상세 패널을 다시 보면, **Then** 빠른 편집 칩과 동일한 값으로 일관되게 반영된다.
4. **Given** 사용자가 목록 카드에서 항목을 보는 상태, **When** 메타데이터 칩 영역을 보면, **Then** 카테고리, 상태, 예약상태, 우선순위 4개 칩이 모두 표시된다.

---

### User Story 4 - 예약 준비 상태를 별도 관리하기 (Priority: P2)

사용자는 "가기로 결정했는지"와 "예약이 필요한지/됐는지"를 구분해 관리할 수 있다.

**Why this priority**: 예약상태 분리는 이번 개편의 구조적 핵심이다. 이 분리가 없으면 기존 상태 필드의 모호함이 그대로 남는다.

**Independent Test**: 동일한 항목에 대해 의사결정 단계와 예약상태를 각각 변경해도 서로 독립적으로 유지되는지 검증하면 된다.

**Acceptance Scenarios**:

1. **Given** 항목이 `검토` 상태인 경우, **When** 사용자가 예약상태를 `불필요`로 설정하면, **Then** 의사결정 단계는 유지되고 예약상태만 변경된다.
2. **Given** 항목이 `확정` 상태이며 예약이 필요한 경우, **When** 사용자가 예약상태를 `예약완료`로 바꾸면, **Then** 상태는 `확정` 그대로 유지되고 예약상태만 갱신된다.
3. **Given** 예약상태 값이 비어 있는 기존 항목, **When** 사용자가 항목을 보면, **Then** "예약 정보 없음" 칩이 표시된다.

## Edge Cases

- 기존 데이터에 없는 신규 카테고리/상태 값이 도입될 때, 과거 저장값과의 매핑이 불완전하면 어떻게 처리할 것인가.
- 예약이 필요 없는 카테고리(예: 산책형 명소)에서도 사용자가 예약상태를 `예약완료`로 설정할 수 있는데, 이를 허용할지 제한할지 결정이 필요하다.
- 드롭다운 저장 중 네트워크 오류가 발생하면, 칩은 이전 값으로 복원되고 재시도 액션이 제공되어야 한다.
- 모바일에서 여러 칩을 빠르게 연속 탭하는 경우, 한 번에 하나의 드롭다운만 열려야 한다.
- placeholder 칩이 많아질 때 카드의 가독성이 과도하게 나빠지지 않도록 우선순위 있는 배치가 필요하다.
- 기존 데이터에 `reservation_status`가 없을 때 `확인 필요`로 보여줄지 placeholder로 보여줄지 마이그레이션 규칙이 필요하다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 기존 `category` 필드를 다음 11개 값으로 제한해야 한다: `교통`, `숙박`, `명소`, `식당`, `카페`, `쇼핑`, `문화시설`, `공연·스포츠`, `액티비티`, `휴양`, `기타`.
- **FR-002**: 시스템은 기존 `status` 필드를 의사결정 단계 필드로 재정의하고 다음 4개 값만 허용해야 한다: `아이디어`, `검토`, `확정`, `제외`.
- **FR-003**: 시스템은 신규 `reservation_status` 필드를 추가하고 다음 4개 값만 허용해야 한다: `확인 필요`, `불필요`, `필요(미예약)`, `예약완료`.
- **FR-004**: 시스템은 `priority` 필드에 다음 3개 값만 허용해야 한다: `반드시`, `들를만해`, `시간 남으면`.
- **FR-005**: 시스템은 목록 카드와 상세 패널에서 `category`, `status`, `reservation_status`, `priority`를 모두 칩 형태로 표시해야 한다.
- **FR-006**: 시스템은 4개 필드 중 값이 비어 있는 경우, 해당 칩을 숨기지 말고 `유형 정보 없음`, `상태 정보 없음`, `예약 정보 없음`, `우선순위 정보 없음`에 준하는 placeholder 칩을 표시해야 한다.
- **FR-007**: 시스템은 사용자가 각 칩을 클릭하면 해당 필드의 옵션 목록이 드롭다운으로 열리도록 해야 한다.
- **FR-008**: 시스템은 드롭다운에서 선택한 값이 즉시 저장되도록 해야 하며, 저장 성공 후 같은 위치의 칩 라벨과 스타일을 갱신해야 한다.
- **FR-009**: 시스템은 저장 중인 칩에 대해 중복 입력을 방지하는 로딩 또는 비활성 상태를 제공해야 한다.
- **FR-010**: 시스템은 드롭다운 외부 클릭 또는 ESC 입력 시 드롭다운을 닫아야 하며, 저장되지 않은 선택 시도는 적용하지 않아야 한다.
- **FR-011**: 시스템은 새 항목 생성 폼과 전체 편집 폼에서도 동일한 필드명과 옵션 체계를 사용해야 한다.
- **FR-012**: 시스템은 기존 데이터에서 새 체계로 넘어올 때 명시된 마이그레이션 규칙에 따라 과거 값을 새 값으로 변환하거나, 변환 불가능한 경우 placeholder 상태로 노출해야 한다.
- **FR-013**: 시스템은 API 검증, 클라이언트 타입, 필터, 정렬, 배지 스타일 정의를 새 필드 체계에 맞게 일관되게 갱신해야 한다.
- **FR-014**: 시스템은 목록 필터와 정렬에서도 새 필드명 및 옵션 체계를 반영해야 한다.
- **FR-015**: 시스템은 `reservation_status`를 데이터 저장소에 영속화해야 한다.
- **FR-016**: 시스템은 목록 카드에서 4개 칩을 모두 항상 표시해야 하며, 칩 영역은 한 줄 래핑을 허용해야 한다.
- **FR-017**: 시스템은 상세 패널에서 4개 칩 전체를 노출하고 각 칩을 클릭 편집 가능하게 해야 한다.
- **FR-018**: 시스템은 카테고리 선택 시 다음 경계 규칙을 사용자 분류 가이드로 사용해야 한다: `박물관/미술관/도서관 -> 문화시설`, `랜드마크/전망대/공원 -> 명소`, `투어/체험/클래스 -> 액티비티`, `스파/온천/휴식 공간 -> 휴양`.

### Key Entities

- **TripItem (확장)**: 여행 항목. 기존 속성에 더해 `category`, `status`, `reservation_status`, `priority`를 새 체계로 보유한다.
- **ItemMetadataOption**: 카테고리, 상태, 예약상태, 우선순위 각 옵션의 값, 표시 라벨, 설명, 톤 스타일을 정의하는 메타데이터.
- **PlaceholderChipState**: 값이 비어 있는 필드를 UI에서 어떤 문구와 스타일로 표시할지 정의하는 상태.

## Migration Rules

### Stored Data Migration Policy

- 기존 저장소 데이터는 새 체계로 마이그레이션 대상이다.
- 마이그레이션은 일회성 변환 스크립트 또는 동등한 관리 작업으로 수행되어야 한다.
- 마이그레이션 이후에는 저장소 기준값도 새 체계만 남도록 정리해야 한다.
- 자동 변환이 불명확한 값은 임의 추정으로 채우지 않고 `null` 또는 `기타`로 보수적으로 처리해야 한다.

### Category Mapping

- `교통 -> 교통`
- `숙소 -> 숙박`
- `식당 -> 식당`
- `카페 -> 카페`
- `관광 -> 명소`
- `공연 -> 공연·스포츠`
- `스포츠 -> 공연·스포츠`
- `쇼핑 -> 쇼핑`
- `기타 -> 기타`

### Status Mapping

- `검토중 -> 검토`
- `보류 -> 아이디어`
- `대기중 -> 검토`
- `확정 -> 확정`
- `탈락 -> 제외`

### Priority Mapping

- `반드시 -> 반드시`
- `들를만해 -> 들를만해`
- `시간 남으면 -> 시간 남으면`

### Reservation Status Initialization

- 기존 데이터에는 `reservation_status`가 없으므로 마이그레이션 시 일괄 기본값을 넣지 않는다.
- 기존 항목의 `reservation_status`는 `null`로 유지한다.
- 마이그레이션 이후 UI에서는 `reservation_status = null`을 `예약 정보 없음` placeholder 칩으로 표시한다.
- 신규 생성 또는 사용자가 직접 수정하는 경우에만 `확인 필요`, `불필요`, `필요(미예약)`, `예약완료` 중 하나를 명시적으로 저장한다.

### Post-Migration Validation

- 마이그레이션 후 저장소에는 더 이상 구 카테고리 값(`숙소`, `관광`, `공연`, `스포츠`)이 남아 있으면 안 된다.
- 마이그레이션 후 저장소에는 더 이상 구 상태 값(`검토중`, `보류`, `대기중`, `탈락`)이 남아 있으면 안 된다.
- 마이그레이션 후 `priority`에는 정의된 3개 값 외의 값이 남아 있으면 안 된다.
- 마이그레이션 후 `reservation_status`는 `null` 또는 정의된 4개 값만 허용된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 목록 카드 또는 상세 패널에서 항목의 4개 메타데이터를 3초 이내에 모두 식별할 수 있다.
- **SC-002**: 상세 패널에서 4개 메타데이터 중 어느 하나라도 전체 편집 폼 진입 없이 2클릭 이내에 변경할 수 있다.
- **SC-003**: 메타데이터 값이 없는 항목도 목록과 상세 패널에서 빈 공간 없이 placeholder 칩으로 100% 표시된다.
- **SC-004**: 새 분류 체계 적용 후 생성/수정/API 검증 경로 간 허용값 불일치가 0건이어야 한다.
- **SC-005**: `reservation_status` 추가 후 사용자는 `확정 여부`와 `예약 완료 여부`를 별도 값으로 관리할 수 있어야 하며, 두 값이 서로 덮어쓰지 않는다.
- **SC-006**: 목록 카드에서 4개 칩이 모두 표시되더라도 모바일 기준 한 카드 내에서 읽기 순서가 유지되고 줄바꿈 시 내용 손실이 없어야 한다.

## Assumptions

- 상세 패널이 가장 먼저 지원해야 하는 빠른 편집 표면이며, 목록 카드의 칩 드롭다운은 후속 확장 가능 범위로 둔다.
- 값이 비어 있는 필드는 저장 시 강제로 기본값을 넣지 않고 nullable 상태를 허용한다. 대신 UI에서 placeholder 칩을 보여준다.
- placeholder 문구는 UX 카피 단계에서 다듬을 수 있으나, 필드별로 서로 구분 가능한 문구여야 한다.
- 기존 `교통/숙소/식당/카페/관광/공연/스포츠/쇼핑/기타` 값은 새 카테고리 체계로 1:1 또는 규칙 기반 매핑 가능하다고 가정한다.
- 기존 `검토중/보류/대기중/확정/탈락` 값은 새 상태 체계로 다음과 같이 매핑한다고 가정한다: `검토중 -> 검토`, `보류 -> 아이디어`, `대기중 -> 검토`, `확정 -> 확정`, `탈락 -> 제외`.
- 기존 데이터에는 `reservation_status`가 없으므로 초기 마이그레이션 시 null을 유지하고, 신규 입력 기본값은 `확인 필요`로 둘 수 있다.
- 목록 카드는 4개 칩 전체 노출을 기본 정책으로 삼고, 상세 패널도 4개 칩 전체 노출 + 클릭 편집을 기본 정책으로 삼는다.

## Clarifications Needed

- **CL-001**: 의사결정 단계의 전이는 자유 변경인지, `아이디어 -> 검토 -> 확정/제외` 같은 추천 흐름을 UI에 반영할지 결정이 필요하다.

## Critical Review / Product Notes

- `스킵` 제거 결정은 타당하다. `제외`와의 중복이 커서 유지했을 때 분류 품질만 나빠질 가능성이 높았다.
- `예약상태`를 별도 필드로 분리한 결정은 타당하다. 다만 `불필요`와 `정보 없음`은 전혀 다른 상태이므로 UI와 데이터에서 명확히 구분해야 한다. `불필요`를 기본값으로 넣으면 예약 필요성이 아직 판단되지 않은 항목이 오염된다.
- 저장소 마이그레이션을 스펙에 포함했으므로, 구현 단계에서는 매핑 정확성 검증과 롤백 가능성 확보가 중요하다. 특히 `관광 -> 명소`, `공연/스포츠 -> 공연·스포츠` 같은 합류 규칙은 데이터 샘플 확인이 필요하다.
- 카테고리 확장은 좋지만 `명소`, `문화시설`, `액티비티`, `휴양`의 경계가 모호할 수 있다. 예를 들어 스파는 `휴양`인지 `액티비티`인지, 박물관은 `명소`인지 `문화시설`인지 규칙이 필요하다.
- "항상 placeholder 칩 표시"는 정보 공백을 드러낸다는 점에서 유효하지만, 카드 밀도를 빠르게 올린다. 카드에서는 필드명 축약, muted 스타일, 한 줄 래핑 규칙까지 함께 설계하지 않으면 시각적 소음이 커질 수 있다.
- 드롭다운 UX는 적절하지만 모바일에서는 바텀시트형 선택 UI가 더 안정적일 수 있다. 이번 스펙은 드롭다운으로 시작하되, 옵션 수가 많은 카테고리는 추후 searchable picker로 커질 여지가 있다.

## Additional Ideas

- 예약상태 칩에만 행동 중심 아이콘을 추가해 `필요(미예약)`과 `예약완료`를 빠르게 구분할 수 있다.
- 목록 필터에서 `확정 + 필요(미예약)` 조합을 빠르게 찾는 saved filter를 제공하면 실제 여행 준비 단계에서 유용하다.
- placeholder 칩을 단순 회색 라벨이 아니라 "설정하기" affordance가 있는 미약한 점선 스타일로 만들면 수정 가능성이 더 잘 읽힌다.
- 상세 패널에서 4개 칩을 한 그룹으로 묶고, 필드 설명을 짧게 노출하면 새 체계 적응 비용을 줄일 수 있다.
